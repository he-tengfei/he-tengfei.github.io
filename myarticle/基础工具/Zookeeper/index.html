Zookeeper

发表于2022-10-26|更新于2022-10-26|基础工具
|字数总计:4.2k|阅读时长:13分钟|阅读量:

简单但是很重要。

一、Zookeeper入门

官网：https://zookeeper.apache.org

1. 概述

Zookeeper是一个开源的分布式的，为分布式应用提供协调服务的Apache项目。(开源代表可以得到源码，分布式代表要在集群里搭建，Apache不一定代表是Apache写的，有可能是公司或个人写的，然后放到Apache托管)

Zookeeper从设计模式角度来理解，是一个基于观察者模式设计的分布式服务管理框架，它负责存储和管理大家都关心的数据，然后接受观察者的注册，一旦这些数据的状态发生了变化，Zookeeper就负责通知已经在Zookeeper上注册的那些观察者做出相应的反应。

Zookeeper = 文件系统 + 通知机制

通过一个小案例了解zookeeper的工作机制：比如想监听三台服务器的在线状态，可以让服务器开启时去zookeeper里注册一下，创建个临时节点，当服务器下线时临时节点也会被删除(注意这里是服务器下线时临时节点就会删除，而不是zookeeper关闭才会删除)。让客户端去监听zookeeper一共有多少个临时节点，临时节点数就是在线服务器数。当有服务器下线时，zookeeper会主动通知各个监听节点，然后监听节点再把新的临时节点数读回来。

zookeeper可以做很多很多事，不要被这个小案例限制住了。

2. 特点

  • Zookeeper：一个领导者（Leader），多个跟随者（Follower）组成的集群
  • 集群中只要有半数以上节点存活，Zookeeper集群就能正常服务(可以有偶数台，但是4台的话，宕机一台，还剩3台，再宕机一台，还剩2台，这2台不满足4的半数以上，就不能正常服务了)
  • 全局数据一致：每个Server保存一份相同的数据副本，因此Client无论连接到哪个Server，数据都是一致的
  • 更新请求顺序进行，来自同一个Client的更新请求按其发送顺序依次执行
  • 数据更新原子性，一次数据更新要么全成功，要么全失败
  • 实时性，在一定时间范围内，Client能读到最新数据

每个zookeeper节点都有一个唯一id，最好是奇数台，有个leader，其余都是follower，可以有很多客户端，并且客户端可以连任何一台zookeeper机器，且读到的数据是一样的。

问题引出：怎么决定谁来当leader呢？为什么半数以上节点存活就能正常服务？怎么保证多台zookeeper节点的数据一致性呢？比如我把数据写到节点1了，那节点2和节点3怎么会有呢？如果把数据在节点1上删除，节点2和节点3会不会也删除呢？

3. 文件系统结构

ZooKeeper数据模型的结构与Unix文件系统很类似，整体上可以看作是一棵树，每个节点称做一个ZNode。每一个ZNode默认能够存储1MB的数据，每个ZNode都可以通过其路径唯一标识。1M够用吗？绝对够用，因为里面存的都是些大伙都关心的数据，比如某些配置项、某个值，一般不会把真正计算的数据往zookeeper里存。znode的名字是我们自己取的。

4. 应用场景

提供的服务包括：统一命名服务、统一配置管理、统一集群管理、服务器节点动态上下线、软负载均衡等。zookeeper在大数据里很重要，但是存在感极低，基本都是配置一下就好了。

统一命名服务：在分布式环境下，经常需要对应用/服务进行统一命名，便于识别。例如：IP不容易记住，而域名容易记住

统一配置管理：分布式环境下，配置文件同步非常常见，一般要求一个集群中，所有节点的配置信息是一致的，比如Kafka集群，对配置文件修改后，希望能够快速同步到各个节点上，配置管理可交由ZooKeeper实现，可将配置信息写入ZooKeeper上的一个Znode，各个客户端服务器监听这个Znode，一旦Znode中的数据被修改，ZooKeeper将通知各个客户端服务器

统一集群管理：分布式环境中，实时掌握每个节点的状态是必要的，可根据节点实时状态做出一些调整。ZooKeeper可以实现实时监控节点状志变化，可将节点信息写入ZooKeeper上的一个ZNode，监听这个ZNode即可获取它的实时状态变化

软负载均衡：在Zookeeper中记录每台服务器的访问数，让访问数最少的服务器去处理最新的客户端请求。

负载均衡：在javaee中一定是要考虑这些问题的，比如京东后台有很多台服务器，你有10个访问请求，这10个访问请求会送给Nginx，它有很多策略，比如轮询。软负载均衡就是通过软件的手段实现负载均衡，比如Nginx，硬负载均衡就是通过硬件的手段实现，但是太贵了一般没人用。所以真正要做负载均衡的话，也是用Nginx来做，不会给zookeeper。

后面可以留意下HA和kafka，分别用zookeeper做了哪些事情。

三、ZK内部原理

1. 节点类型

持久（Persistent）：客户端和服务器端断开连接后，创建的节点不删除

  • 持久化目录节点，客户端与Zookeeper断开连接后，该节点依旧存在
  • 持久化顺序编号目录节点，客户端与Zookeeper断开连接后，该节点依旧存在，只是Zookeeper给该节点名称进行顺序编号，比如znode2_001，znode2是自己起的，后面的001是系统生成的，由父节点维护

短暂（Ephemeral）：客户端和服务器端断开连接后，创建的节点会被删除

  • 临时目录节点，客户端与Zookeeper断开连接后，该节点被删除
  • 临时顺序编号目录节点，客户端与Zookeeper断开连接后，该节点被删除，只是Zokeeper给该节点名称进行顺序编号，注意编号是不会往后退的，比如你把002的结构体删了，下一个结构体的编号依然是003，临时节点不能有子节点

说明：创建znodel时设置顺序标识，znode名称后会附加一个值，顺序号是一个单调递增的计数器，由父节点维护

注意：在分布式系统中，顺序号可以被用于为所有的事件进行全局排序，这样客户端可以通过顺序号推断事件的顺序

2. Stat结构体

  • czxid：创建节点的事务zxid，每次修改ZooKeeper状态都会收到一个zxid形式的时间戳，也就是ZooKeeper事务ID。事务ID是ZooKeeper中所有修改总的次序。每个修改都有唯一的zxid，如果zxid1小于zxid2，那么zxid1在zxid2之前发生。选举leader时会用到这个，zxid大的说明数据较完整
  • ctime：znode被创建的毫秒数(从1970年开始)
  • mzxid：znode最后更新的事务zxid
  • mtime：znode最后修改的毫秒数(从1970年开始)
  • pZxid：znode最后更新的子节点zxid
  • cversion：znode子节点变化号，znode子节点修改次数
  • dataversion：znode数据变化号，数据版本号
  • aclVersion：znode访问控制列表的变化号
  • ephemeralOwner：如果是临时节点，这个是znode拥有者的session id，如果不是临时节点则是0
  • dataLength：znode的数据长度
  • numChildren：znode子节点数量

3. 监听器原理(重点)

监听原理详解：首先要有一个main线程，在main线程中创建Zookeeper客户端，这时就会创建两个线程，一个负责网络连接通信connet，一个负责监听listener，通过connect线程将注册的监听事件发送给Zookeeper，在Zookeeper的注册监听器列表中将注册的监听事件添加到列表中(就是记录了哪个人对哪个数据感兴趣)，当Zookeeper监听到有数据或路径变化，就会将这个消息发送给对它感兴趣的listener线程，listener线程内部调用了process方法，用户自己在这个方法里写要做什么事情，比如重新读一次监听列表什么的。(上面的listener、connect线程在zookeeper.java类里能找到，叫sendthread和enventthread)

常见的监听：监听节点数据的变化、监听子节点增减的变化

5. 选举机制(重点)

ZK是基于ZAB协议进行选举的，ZAB协议可以看做paxos的升级版，是ZK自己搞的一套协议，它主要做了两件事情：没有leader的时候选举leader，有leader的时候保证数据一致性。

半数机制：集群中半数以上机器存活，集群可用，所以Zookeeper适合安装奇数台服务器

leader与follower：Zookeeper虽然在配置文件中并没有指定Master和Slave。但是，Zookeeper工作时，是有一个节点为Leader，其他则为Follower，Leader是通过内部的选举机制临时产生的。

以一个简单的例子来说明整个选举的过程：

假设有五台服务器组成的Zookeeper集群，它们的id从1-5，同时它们都是最新启动的，也就是没有历史数据，在存放数据量这一点上，都是一样的。假设这些服务器依序启动，来看看会发生什么。

  • 服务器1启动，发起一次选举。服务器1投自己一票。此时服务器1票数一票，不够半数以上（3票），选举无法完成，服务器1状态保持为LOOKING
  • 服务器2启动，再发起一次选举。服务器1和2分别投自己一票并交换选票信息：此时服务器1发现服务器2的ID比自己目前投票推举的（服务器1）大，更改选票为推举服务器2。此时服务器1票数0票，服务器2票数2票，没有半数以上结果，选举无法完成，服务器1，2状态保持LOOKING
  • 服务器3启动，发起一次选举。此时服务器1和2都会更改选票为服务器3。此次投票结果：服务器1为0票，服务器2为0票，服务器3为3票。此时服务器3的票数已经超过半数，服务器3当选Leader。服务器1，2更改状态为FOLLOWING，服务器3更改状态为LEADING
  • 服务器4启动，发起一次选举。此时服务器1，2，3已经不是LOOKING状态，不会更改选票信息。交换选票信息结果：服务器3为3票，服务器4为1票。此时服务器4服从多数，更改选票信息为服务器3，并更改状态为FOLLOWING
  • 服务器5启动，同4一样当小弟

集群启动的选举过程(集群刚搭建好的时候，zxid一致，例如都为0)

  • 假设集群有5台机器,每台机器有自己的唯一编号(myid)，这个编号是用户自定义的，假设分别是12345，且机器启动的顺序为12345

  • 先启动机器1(myid为1)，为自己投票(zxid,myid)，实际投的票为(0,1)，因为机器没有达到半数以上，选举失败

  • 启动机器2(myid为2)，为自己投票(zxid,myid)，实际投的票为(0,2)，机器1和机器2交换选票，互相看看对方投的谁。此刻机器1得到的投票信息为(0,1) (0,2)，这里比的是myid，机器1发现机器2的投票信息的myid比自己的大，机器1也改投(0,2)。同理机器2维持当前自己的投票(0,2)。最终，机器2得到两票，但是因为没有达到半数以上，选举失败

  • 启动机器3(myid为3)，每次投票都是重新投，机器1的投票(0,1)，机器2的投票(0,2)，机器3的投票(0,3)，相互交换投票，同理机器3得到3票。因为已经满足半数以上，机器3当选为leader，机器1和机器2服从机器3的领导，作为follower

  • 机器4和机器5启动，因为已经选举出leader，只能作为follower

  • 如果启动顺序是54321，那么5就是leader了，因为zxid相同的话，比的是myid

leader故障后的选举过程

往ZK写数据是通过leader写的，leader会把要写的操作广播给follower，follower要告诉leader同不同意写这个数据，然后leader通知follower开始写，刚通知完节点1，leader突然挂了，这样节点1就会正常写数，别的节点没收到通知就不会写(理论上这种情况可能发生的概率跟中彩票一样小)。这时重新投票选leader，节点1的zxid肯定是最大的，每台机器都发出投票(zxid,myid)，交换选票，会先比较zxid，如果自己投票中的zxid小于别人的投票的zxid，则改投zxid大的，最终zxid大的机器会当选leader，如果每台机器的zxid都一致，myid大的当选leader。

6. 写数据流程

写请求必须通过leader来广播，读请求无所谓

  • Client向ZooKeeper的Server1上写数据，发送一个写请求
  • 如果Server1不是Leader，那么Server1会把接收到的请求进一步转发给Leader，因为每个ZooKeeper的Server里面有一个是Leader。这个Leader会将写请求广播给各个Server，比如Server1和Server2，各个Server会将该写请求加入待写队列，并向Leader发送成功信息
  • 当Leader收到半数以上Server的成功信息，说明该写操作可以执行。Leader会向各个Server 发送提交信息，各个Server收到信息后会落实队列里的写请求，此时写成功的话会再通知leader，当leader收到半数以上的成功信息，会通知给server1(因为客户端连接的是server1)
  • Server1会进一步通知Client数据写成功了，这时就认为整个写操作成功

*待写队列，队列就是先进先出的，所以可以理解前面说的ZK的写请求是按时间顺序执行的。半数以上就行，不发消息的就不管它了

问题：半数以上就行的话，是怎么保证ZK各节点之间的数据一致性的呢？还有前面的leader选举，follower如何保证跟leader的数据一致性呢？

四、ZK实操

命令基本语法 功能描述
help 显示所有操作命令
ls path 使用 ls 命令来查看当前znode的子节点 [可监听] -w 监听子节点变化 -s 附加次级信息 ，注意这里的监听，只管一次，第二次发生变化就监听不到了
create 普通创建 -s 含有序列 -e 临时（重启或者超时消失），不写参数就是永久节点
get path 获得节点的值 [可监听] -w 监听节点内容变化 -s 附加次级信息，也是只管一次
set 设置节点的具体值，这是跟Linux目录相似又不同于目录的地方
stat 查看节点状态，也可以用ls -s，相当于ls + stat
delete 删除节点
deleteall 递归删除节点

十一、Table API 和SQL

在Flink提供的多层级API中，核心是DataStreamAPI，底层是处理函数process function，可以访问事件的时间信息、注册定时器、自定义状态，进行有状态的流处理。DataStreamAPI和处理函数比较通用，有了这些API，理论上我们就可以实现所有场景的需求了。

不过企业一般会将底层API包装成更加具体的应用级接口。怎样的接口风格最容易让大家接收呢？当然是SQL了。SQL是结构化查询语言Structured Query Language的缩写，是我们对关系型数据库进行查询和修改的通用编程语言。Flink同样提供了对于表处理的支持，这就是更高层级的应用API，在Flink中被称为TableAPI和SQL。

在Flink中，这两种API被集成在一起，SQL执行的对象也是Flink中的表Table，所以我们一般会认为它们是一体的。TableAPI和SQL最初并不完善，在Flink1.9版本合并阿里巴巴内部版本Blink之后发生了非常大的改变，而即使是在目前最新的1.13版本中，TableAPI和SQL也依然不算稳定，接口用法还在不停调整和更新。所以这部分希望大家重在理解原理和基本用法，具体的API调用可以随时关注官网的更新变化。

1. 快速上手

Table API 和SQL 的使用其实非常简单，只要得到一个表Table，然后对它调用 Table API，或者直接写 SQL 就可以了。

2. 基本API

Table API 和 SQL 可以看作联结在一起的一套 API，这套 API 的核心概念就是表Table。输入数据可以定义成一张表，然后对这张表进行查询，就可以得到新的表，最后还可以定义一张用于输出的表，将处理结果写入到外部系统。

2.1 程序基本结构

这里的CREATE语句中用WITH指定了外部系统的连接器，于是就可以连接外部系统读写数据了。基于这种程序架构，就可以完全抛开DataStreamAPI，直接用SQL语句实现全部的流处理过程。在创建表的过程中，其实并不区分输入还是输出，只需要将这个表注册进来，连接到外部系统就可以了

2.2 创建表环境

每个表和SQL的执行，都必须绑定在一个表环境中。TableEnvironment是TableAPI中提供的基本接口类，可以通过调用静态的create()方法来创建一个表环境实例。方法需要传入一个环境的配置参数EnvironmentSettings，它可以指定当前表环境的执行模式和计划器（planner）。执行模式有批处理和流处理两种选择，默认是流处理模式，计划器默认使用blinkplanner。

对于流处理场景，默认配置就完全够用，可以用另一种更加简单的方式来创建表环境， StreamTableEnvironment 是继承自TableEnvironment 的子接口。调用它的 create() 方法， 只需要直接将当前的流式执行环境传入，就可以创建出对应的流式表环境了。

TableEnvironment主要负责：

  • 注册Catalog和表
  • 执行SQL查询
  • 注册用户自定义函数（UDF）
  • DataStream和表之间的转换

这里的Catalog就是目录，主要用来管理所有数据库和表的元数据metadata，可以认为我们所定义的表都会挂靠在某个目录下，这样就可以快速检索。在表环境中可以由用户自定义Catalog，并在其中注册表和自定义函数UDF，默认的Catalog叫作default_catalog。

2.3 创建表

为了方便地查询表，表环境中会维护一个目录Catalog和表的对应关系，所以表都是通过Catalog注册创建的。表在环境中有一个唯一的ID，由三部分组成：目录名，数据库名，以及表名。在默认情况下，目录名为default_catalog，数据库名为default_database。所以如果我们直接创建一个叫作MyTable的表，它的ID就是：default_catalog.default_database.MyTable。

具体创建表的方式，有通过连接器connector和虚拟表virtual tables两种。

2.3.1 连接器表 Connector Tables

通过WITH关键字指定连接到外部系统的连接器，这里的TEMPORARY关键字可以省略。连接器会在11.8节中展开讲解。这里没有定义Catalog和Database，所以都是默认的，表的完整ID就是default_catalog.default_database.MyTable。如果希望使用自定义的目录名和库名，可以在环境中进行设置，这样我们创建的表完整ID就变成了custom_catalog.custom_database.MyTable。之后在表环境中创建的所有表，ID也会都以custom_catalog.custom_database作为前缀。

2.4 表的查询

Flink为我们提供了两种查询方式：SQL和Table API。

2.4.1 执行SQL查询

Flink基于Apache Calcite来提供对SQL的支持，Calcite是一个为不同的计算平台提供标准SQL查询的底层工具，很多大数据框架比如ApacheHive、ApacheKylin中的SQL支持都是通过集成Calcite来实现的。

只要调用表环境的sqlQuery()方法，传入一条SQL就可以了，执行得到的结果，是一个Table对象。目前 Flink 支持标准 SQL 中的绝大部分用法，并提供了丰富的计算函数。

2.4.2 调用Table API查询

通过一步步链式调用Table的方法，就可以定义出所有的查询转换操作，每一步方法调用的返回结果，都是一个Table。

由于Table API是基于Table的Java实例进行调用的，因此我们首先要得到表的Java对象。基于环境中已注册的表，可以通过表环境的from()方法非常容易地得到一个Table对象：Table eventTable=tableEnv.from("EventTable")，传入的参数就是注册好的表名。得到Table对象之后，就可以调用API进行各种转换操作了：Table result2 = eventTable.select($("url"), $("user"))。

Table API的方式基本不会用，以后也是往SQL层面发展。

2.4.3 两种API的结合使用

无论是调用TableAPI还是执行SQL，得到的结果都是一个Table对象，所以这两种API的查询可以很方便地结合在一起。无论是那种方式得到的Table对象，都可以继续调用Table API进行查询转换。如果想要对一个表执行SQL操作，必须先在环境中对它进行注册。所以我们可以通过创建虚拟表的方式实现两者的转换：tableEnv.createTemporaryView(“MyTable”,myTable)，前面那个是自定义表名，后面那个是表对象。

之前我们并没有将Table对象注册为虚拟表就直接在SQL中使用了，这其实是一种简略的写法，我们将Table对象名eventTable直接以字符串拼接的形式添加到SQL语句中，在解析时会自动注册一个同名的虚拟表到环境中，这样就省略了创建虚拟视图的步骤。

尽量不要结合使用，企业项目中往往会直接选择SQL的方式来实现需求。

2.5 输出表

输出一张表最直接的方法，就是调用Table的方法executeInsert()将一个Table写入到注册过的表中，方法传入的参数就是注册的表名：resultTable.executeInsert("output")。这行的意思是，把结果表的数据写入output表，output表在创建的时候有个connector，里面应该有对TableSink的实现。

在底层，表的输出是通过将数据写入到TableSink来实现的。TableSink是TableAPI中提供的一个向外部系统写入数据的通用接口，可以支持不同的文件格式（比如CSV、Parquet）、存储数据库（比如JDBC、HBase、Elasticsearch）和消息队列（比如Kafka）。它有些类似于DataStreamAPI中调用addSink()方法时传入的SinkFunction，有不同的连接器对它进行了实现。

(在早期的版本中，有专门用于输入输出的TableSource和TableSink，但是这种方式与SQL的使用习惯不符，已弃用。不知道这里说的TableSink和上面说的TableSink是不是一个东西。)

2.6 表和流的转换

测试业务逻辑一般不会直接将结果直接写入到外部系统，而是在本地控制台打印输出。对于DataStream这非常容易，直接调用print()方法就可以了，但对于Table就比较悲剧，它没有提供print()方法。这该怎么办呢？我们可以将Table再转换成DataStream，然后进行打印输出，这就涉及了表和流的转换。

2.6.1 将表转换成流

toDataStream()和toChangelogStream()，前者不支持表的update操作，对于这样有更新操作的表，我们不要试图直接把它转换成 DataStream 打印输出，而是记录一下它的更新日志change log。这样一来，对于表的所有更新操作，就变成了一条更新日志的流，我们就可以转换成流打印输出了。所以前者又叫仅插入流Insert-Only Streams，后者叫更新日志流Changelog Streams，具体使用哪种方式取决于表中是否存在更新（update）操作。

2.6.2 将流转换成表

可以直接调用fromDataStream()方法，由于流中的数据是POJO类，所以将流转换成表之后，每一行数据就对应着一个Event，而表中的列名就对应着Event中的属性。我们还可以在fromDataStream()方法中给列名重命名或者改变位置，比如把关键字timestamp重命名为ts。

通过fromDataStream()转换成的表不能直接在SQL里用，还要另外注册。可以用createTemporaryView直接创建成虚拟视图。

另外表环境还提供了一个fromChangelogStream()方法，可以将一个更新日志流转换成表。这个方法要求流中的数据类型只能是 Row，而且每一个数据都需要指定当前行的更新类型RowKind，所以一般是由连接器帮我们实现的，直接应用比较少见，感兴趣的读者可以查看官网的文档说明。

2.6.3 支持的数据类型

DataStream支持的数据类型，Table也都支持，大概有基本数据类型、POJO类型和Row类型，并且基础数据类型默认的字段名是f0，这是把原子类型当作了一元组Tuple1。需要特别注意的是Row类型，用起来不方便，需要指定RowKind，用来表示当前行在更新操作中的类型，所以好像只在connector里用。

3. 流处理中的表

3.1 动态表和持续查询

动态表：对于流式数据，当流中有新数据到来，表中会插入一行，这样得到的表就会不断地动态变化，被称为动态表DynamicTables。在很多高级关系型数据库（比如Oracle、DB2）中都有物化视图Materialized Views的概念，可以用来缓存SQL查询的结果，它的更新其实就是不停地处理更新日志流的过程。Flink中的动态表，就借鉴了物化视图的思想。

持续查询：对动态表定义的查询操作，都是持续查询，而持续查询的结果也会是一个动态表。每次数据到来都会触发查询操作，每次查询面对的数据集，就是当前输入动态表中收到的所有数据。

持续查询的步骤如下：

  • 流被转换为动态表
  • 对动态表进行持续查询，生成新的动态表
  • 生成的动态表被转换成流

这样，只要将流和动态表的转换封装起来，我们就可以直接在数据流上执行 SQL 查询，用处理表的方式来做流处理了。

3.2 用 SQL 持续查询

当原始动态表不停地插入新的数据时，查询得到的urlCountTable会持续地进行更改，可能是插入，也可能是更新。换句话说，用来定义结果表的更新日志流中，包含了INSERT和UPDATE两种操作。这种持续查询被称为更新查询UpdateQuery，更新查询得到的结果表如果想要转换成DataStream，必须调用toChangelogStream()方法。而更新日志流中只有插入的持续查询，被称为追加查询AppendQuery，追加查询得到的结果表，转换成DataStream调用的方法没有限制，可以直接用toDataStream()，也可以像更新查询一样调用toChangelogStream()。

什么时候表的结果只有插入呢？简单的select查询，或者基于窗口的聚合查询。窗口如果设置了允许迟到，结果也是一次性写入结果表吗？

持续查询有个查询限制的问题，有些持续查询会因为计算代价太高而受到限制。比如对于rank()函数，每来一条新的数据，更新结果的时候可能需要全部重新计算，并且对很多已经输出的行进行更新。这样的查询操作，就不太适合作为连续查询在流处理中执行。或者给RANK()配合一个OVER 子句，这是所谓的开窗聚合。

3.3 表转流的编码方式

动态表也可以增删改，可以通过编码后的消息指明所有的增删改操作。在 Flink 中，Table API 和 SQL 支持三种编码方式：

  • 仅追加流Append-only：只有插入
  • 撤回流Retract：包括之前数据的删除和新数据的插入
  • 更新插入流Upsert：需要动态表中有唯一的键key，通过这个key进行查询，如果存在对应的数据就更新，不存在就插入。更新插入流跟撤回流的主要区别在于，更新操作由于有key的存在，只需要用单条消息编码就可以，因此效率更高

在代码里将动态表转换为DataStream时，只支持仅追加和撤回流，调用toChangelogStream()得到的就是撤回流。DataStream中并没有key的定义，所以只能通过两条消息一减一增来表示更新操作，而连接到外部系统时，则可以支持不同的编码方式。

4. 时间属性和窗口

在 Table API 和 SQL 中，会给表单独提供一个时间字段，专门用来在表处理程序中指示时间。它可以在创建表的DDL 里直接定义为一个字段，也可以在 DataStream 转换成表时定义。时间属性的数据类型为TIMESTAMP，它的行为类似于常规时间戳，可以直接访问并且进行计算。

4.1 事件时间

4.1.1 在创建表时定义

可以增加一个字段，通过 WATERMARK 语句来定义事件时间属性。这里我们把 ts 字段定义为事件时间属性，而且基于 ts 设置了 5 秒的水位线延迟，这里的5 秒是以时间间隔的形式定义的，数值必须用单引号引起来。

事件时间的数据类型必须为TIMESTAMP或者TIMESTAMP_LTZ。TIMESTAMP_LTZ是指带有本地时区信息的时间戳，一般情况下如果数据中的时间戳是“年-月-日-时-分-秒”的形式，那就是不带时区信息的，可以将事件时间属性定义为TIMESTAMP类型。而如果原始的时间戳就是一个长整型的毫秒数，类型定义为TIMESTAMP_LTZ会更方便：

这里我们另外定义了一个字段ts_ltz，是把长整型的ts转换为TIMESTAMP_LTZ得到的，进而使用WATERMARK语句将它设为事件时间属性，并设置5秒的水位线延迟。

4.1.2 在流转换为表时定义

流转表时，给某个字段加上.rowtime()后缀，就表示将当前字段指定为事件时间属性。这个字段可以是数据中本不存在，额外追加上去的逻辑字段，也可以是本身固有的字段，那么这个字段就会被事件时间属性所覆盖，类型也会被转换为TIMESTAMP。不论那种方式，事件时间属性字段中保存的都是TIMESTAMP类型。这种方式只负责指定时间属性，而时间戳的提取和水位线的生成应该之前就在DataStream上定义好了。由于DataStream中没有时区概念，因此Flink会将事件时间属性解析成不带时区的TIMESTAMP类型，所有的时间都被当作UTC标准时间。

4.2 处理时间

相比之下处理时间就比较简单了，它就是我们的系统时间，使用时不需要提取时间戳和生成水位线，只需额外声明一个字段，专门用来保存当前的处理时间。

4.2.1 在创建表时定义

在创建表时，增加一个额外的字段，调用系统内置的PROCTIME()函数来指定处理时间，返回的类型是TIMESTAMP_LTZ。这里的时间属性，其实是以计算列的形式定义出来的，用一个AS语句来在表中产生数据中不存在的列，并且可以利用原有的列、各种运算符及内置函数。在前面事件时间属性的定义中，将ts字段转换成TIMESTAMP_LTZ类型的ts_ltz，也是计算列的定义方式。

4.2.2 在流转换为表时定义

由于处理时间是系统时间，原始数据中并没有这个字段，所以处理时间属性一定不能定义在一个已有字段上，只能定义在表结构所有字段的最后，作为额外的逻辑字段出现。

4.3 窗口

4.3.1 分组窗口

在Flink1.12之前的版本中，TableAPI和SQL提供了一组分组窗口函数（Group Window），分组窗口的功能比较有限，目前已经处于弃用（deprecated）的状态。

4.3.2 窗口表值函数

从1.13版本开始，Flink开始使用窗口表值函数（Windowing table-valued functions，WindowingTVFs）来定义窗口。表函数（table function）可以看作是返回一个表的函数，后面会讲。目前Flink提供了以下几个窗口TVF：

  • 滚动窗口（TumblingWindows）
  • 滑动窗口（HopWindows）
  • 累积窗口（CumulateWindows）
  • 会话窗口（SessionWindows，目前尚未完全支持）

目前窗口TVF的功能还不完善，会话窗口和很多高级功能还不支持，不过正在快速地更新完善。可以预见在未来的版本中，窗口TVF将越来越强大，将会是窗口处理的唯一入口。

在窗口TVF的返回值中，除去原始表中的所有列，还增加了用来描述窗口的额外3个列：窗口起始点window_start、窗口结束点window_end、窗口时间window_time。这里的窗口时间等于window_end-1ms，所以相当于是窗口中能够包含数据的最大时间戳。

在SQL中的声明方式，直接调用TUMBLE()、HOP()、CUMULATE()就可以实现滚动、滑动和累积窗口，不过传入的参数会有所不同。下面我们就分别对这几种窗口TVF进行介绍。

(1) 滚动窗口

滚动窗口只有一个核心参数就是窗口大小，暂不支持计数窗口，所以滚动窗口就是滚动时间窗口。这里基于时间字段ts，对表EventTable中的数据开了大小为1小时的滚动窗口。窗口会将表中的每一行数据，按照它们ts的值分配到一个指定的窗口中。

(2) 滑动窗口

通过调用HOP()来声明滑动窗口，除了要传入表名、时间属性外，还需要传入滑动步长和窗口大小两个参数。这里基于时间属性ts，在表EventTable上创建了大小为1小时的滑动窗口，每5分钟滑动一次。

(3) 累积窗口

有这样一类需求：我们的统计周期可能较长，因此希望中间每隔一段时间就输出一次当前的统计值。例如，我们按天来统计网站的PV，如果用1天的滚动窗口，那需要到每天24点才会计算一次，输出频率太低，如果用滑动窗口，计算频率可以更高，但统计的就变成了“过去24小时的PV”。我们真正希望的是，按照自然日统计每天的PV，不过每隔1小时就输出一次当天到目前为止的PV值。这种特殊的窗口就叫作累积窗口Cumulate Window，累积窗口中有两个核心的参数：累积步长和最大窗口长度。

这里我们基于时间属性ts，在表EventTable上定义了一个统计周期为1天、累积步长为1小时的累积窗口。

到此为止，上面只是定义了窗口，窗口的完整调用包括开窗+聚合两步。

6. 函数

Flink 的Table API 和 SQL 同样提供了函数的功能，两者在调用时略有不同：Table API 是通过方法调用来实现的，而 SQL 则是直接引用函数名称，传入数据作为参数。例如，要把一个字符串 str 转换成全大写的形式，Table API 的写法是str.upperCase()，而 SQL 中的写法是UPPER(str)。主要还是 SQL 中函数的使用。

Flink SQL中的函数可以分为两类：一类是SQL中内置的系统函数，比如COUNT()、CHAR_LENGTH()、UPPER()等等，而另一类是用户自定义的函数UDF，需要在表环境中注册才能使用。

这部分主要是一个知识扩展，实际应用的场景较少，一般只需要知道系统函数的用法就够了。

6.1 系统函数

系统函数也叫内置函数，FlinkSQL中的系统函数分为两大类：标量函数ScalarFunctions和聚合函数AggregateFunctions。

标量函数：+ - * / =，POWER，RAND，CHAR_LENGTH，TIMESTAMP，CURRENT_TIME

聚合函数：COUNT()、SUM()、RANK()、ROW_NUMBER()

6.2 自定义函数 UDF

Flink 的Table API 和SQL 提供了多种自定义函数的接口，以抽象类的形式定义。当前 UDF主要有以下几类：

  • 标量函数 Scalar Functions ：将输入的标量值转换成一个新的标量值
  • 表函数 Table Functions ：将标量值转换成一个或多个新的行数据，也就是扩展成一个表
  • 聚合函数 Aggregate Functions ：将多行数据里的标量值转换成一个新的标量值
  • 表聚合函数 Table Aggregate Functions ：将多行数据里的标量值转换成一个或多个新的行数据

6.2.1 自定义函数的调用流程

首先自定义对应 UDF 抽象类的实现，并在表环境中注册这个函数，然后就可以在 Table API 和 SQL 中调用了。

6.2.2 标量函数

想要实现自定义的标量函数，我们需要自定义一个类来继承抽象类 ScalarFunction，并实现叫作eval() 的求值方法。它必须是公有的（public），而且名字必须是 eval。求值方法 eval 可以重载多次，任何数据类型都可作为求值方法的参数和返回值类型。ScalarFunction 抽象类中并没有定义 eval()方法，所以我们不能直接在代码中重写（override），但 Table API 的框架底层又要求了求值方法必须名字为 eval()。这是 Table API 和 SQL 目前还显得不够完善的地方，未来的版本应该会有所改进。ScalarFunction 以及其它所有的UDF 接口，都在 org.apache.flink.table.functions 中。

6.2.4 聚合函数

多对一，类似于UDAF。自定义聚合函数需要继承抽象类AggregateFunction。它有两个泛型参数<T, ACC>，T表示聚合输出的结果类型，ACC则表示聚合的中间状态类型。聚合函数的所有方法都必须是公用的。

聚合函数的工作原理如下：

  • 首先它需要调用createAccumulator()创建一个空的累加器，用来存储聚合的中间结果
  • 对于输入的每一行数据，都会调用accumulate()方法来更新累加器，这是聚合的核心过程。注意它的第一个参数是累加器，之后的参数是调用函数时传入的形参。不能重写只能手敲
  • 当所有的数据都处理完之后，通过调用getValue()方法来计算并返回最终的结果。在遇到复杂类型时，Flink的类型推导可能会无法得到正确的结果。所以AggregateFunction也可以专门对累加器和返回结果的类型进行声明，这是通过getAccumulatorType()和getResultType()两个方法来指定的

十二、Flink CEP

在实际应用中，还有一类需求是要检测以特定顺序先后发生的一组事件，进行统计或做报警提示。例如检测连续登录失败事件。类似的多个事件的组合，我们把它叫作复杂事件。对于复杂时间的处理，由于涉及到事件的严格顺序，有时还有时间约束，我们很难直接用 SQL 或者 DataStream API 来完成。处理函数确实可以搞定这些需求，但是复杂度会非常高，很可能会使代码失去可读性。于是Flink 为我们提供了专门用于处理复杂事件的库——CEP，可以让我们更加轻松地解决这类棘手的问题。这在企业的实时风险控制中有非常重要的作用。

1. CEP与模式

所谓CEP，就是复杂事件处理Complex Event Processing的缩写。Flink CEP，就是 Flink 实现的一个用于复杂事件处理的库。可以在事件流里，检测到特定的事件组合并进行处理，比如说连续登录失败，或者订单支付超时等等。

CEP的流程可以分成三个步骤：

  • 定义一个匹配规则，又叫模式pattern
  • 将匹配规则应用到事件流上，检测满足规则的复杂事件
  • 对检测到的复杂事件进行处理，得到结果进行输出

CEP的第一步所定义的匹配规则，我们可以把它叫作模式Pattern。模式的定义主要就是两部分内容：

  • 每个简单事件的特征
  • 简单事件之间的组合关系，比如事件的发生顺序，顺序又分严格近邻宽松近邻等，前后两个事件的时间间隔，事件是否可以重复出现等等

Flink CEP为我们提供了丰富的API，可以实现上面关于模式的所有功能，这套API就叫作模式API，一般的应用场景大概在风险控制、用户画像、运维监控等方面。很多大数据框架如Spark都提供了不同的CEP解决方案，但没有专门的库，而Flink提供了专门的CEP库用于复杂事件处理，可以说是目前CEP的最佳解决方案。

2. 快速上手

为了精简和避免依赖冲突，Flink会保持尽量少的核心依赖，所以核心依赖中并不包括任何的连接器和库，这里的库就包括了SQL、CEP等等。所以如果想要在Flink集群中提交运行CEP作业，应该向Flink SQL那样将依赖的jar包放在/lib目录下。从这个角度来看，Flink CEP和Flink SQL一样，都是最顶层的应用级API。

3. Pattern API

3.1 个体模式

每个简单事件的匹配规则，叫作个体模式。每个个体模式都以一个连接词开始，比如begin、next等等，这是Pattern对象的一个方法，返回的还是一个Pattern。这些连接词方法有一个String类型参数，这就是当前个体模式唯一的名字，在之后检测到匹配事件时，就会以这个名字来指代匹配事件。

个体模式需要一个过滤条件，用来指定具体的匹配规则，这个条件一般通过where()方法实现，在个体模式后面加个量词就能匹配多次，还可以指定严格近邻、贪婪匹配、匹配一次或多次、可选匹配等。之前代码中没有定义量词，都是单例模式，所以只会匹配一个事件，每个List 中也只有一个元素。

3.2 组合模式

组合模式确实就是一个模式序列，是用诸如 begin、next、followedBy 等表示先后顺序的连接词将个体模式串连起来得到的。组合模式的近邻条件又分为严格近邻、宽松近邻、非确定性宽松近邻等，分别对应next()、followedBy()、followedByAny()方法。或者调用within()指定一个模式序列必须在多长时间内匹配完成。在有量词的事件里，可以用consecutive()和allowCombinations()来指定严格模式和非确定性宽松近邻，默认是宽松模式。

3.3 模式组

多个模式序列串起来就是模式组。

3.4 匹配后跳过策略

对(a1 a2 a3 b)去匹配b，那么应该检测到 6 个匹配结果：(a1 a2 a3 b)，(a1 a2 b)，(a1 b)，(a2 a3 b)，(a2 b)，(a3 b)。匹配的太多了，可以自定义只要(a1 a2 a3 b)别的跳过，或者只要(a1 a2 a3 b)，(a2 a3 b)，(a3 b)别的跳过。

PatternStream 的处理主要可以分成两种：简单便捷的选择提取操作select，和更加通用更加强大的处理操作process。选择操作传入的是一个 PatternSelectFunction或PatternFlatSelectFunction，处理操作传入的则是一个 PatternProcessFunction，后者是目前官方推荐的处理方式。事实上，PatternSelectFunction 和 PatternFlatSelectFunction 在 CEP 内部执行时也会被转换成 PatternProcessFunction。

4.3 处理超时事件

一般来说，符合模式匹配的事件，会保存到对应的 Map 中，不符合的会丢弃。但是within()不同，在开头能够正常匹配的前提下，没有等到后续的匹配事件才会超时，所以往往不应该直接丢弃，这部分数据可以捕获，并由侧输出流输出。比如订单的超时检测用这个就很方便，它可以拿到哪单超时未支付，并给出这单的相关信息。用处理函数进行状态编程，结合定时器也可以实现同样的功能，但明显 CEP 的实现更加方便， 也更容易迁移和扩展。

4.4 处理迟到数据

在 Flink CEP 中沿用了通过设置水位线（watermark）延迟来处理乱序数据的做法。当一个事件到来时，并不会立即做检测匹配处理，而是先放入一个缓冲区。缓冲区内的数据，会按照时间戳由小到大排序，当一个水位线到来时，就会将缓冲区中所有时间戳小于水位线的事件依次取出，进行检测匹配。如果有数据再迟到又不想丢弃，也可以将迟到事件输出到侧输出流。

5. CEP 的状态机实现

CEP 中模式的定义方式比较复杂，与正则表达式非常相似，它俩都是用状态机实现的。下面以检测连续3次登录失败为例，说下状态机的工作原理。

从初始状态INITIAL出发，遇到一个类型为fail 的登录失败事件，就开始进入部分匹配的状态。目前只有一个 fail 事件，我们把当前状态记作 S1。基于S1如果继续遇到 fail 事件，那么就有两个 fail 事件，记作 S2。基于S2状态如果再次遇到 fail 事件，那么就找到了一组匹配的复杂事件，把当前状态记作 Matched， 就可以输出报警信息了。需要注意的是，报警完毕，需要立即重置状态回 S2，因为如果接下来再遇到 fail 事件，就又满足了新的连续三次登录失败，需要再次报警。

而不论是初始状态，还是 S1、S2 状态，只要遇到类型为 success 的登录成功事件，就会跳转到结束状态，记作 Terminal。此时当前检测完毕，之前的部分匹配应该全部清空，所以需要立即重置状态到 Initial，重新开始下一轮检测。所以这里我们真正参与状态转移的，其实只有 Initial、S1、S2 三个状态，Matched 和Terminal 是为了方便我们做其他操作（比如输出报警、清空状态）的临时标记状态，不等新事件到来马上就会跳转。

状态机的实现非常复杂，所以 Flink CEP 将底层状态机NFA全部实现好并封装起来，这样我们处理复杂事件时只要调上层的 Pattern API 就可以， 无疑大大降低了代码的复杂度，提高了编程的效率

6. 本章总结

DataStream API 和处理函数是 Flink 应用的基石，SQL 和CEP 是 Flink 大厦顶层扩展的两大工具。CEP 在实际生产中有非常广泛的应用。对于大数据分析而言，应用场景主要可以分为统计分析和逻辑分析。企业的报表统计、商业决策都离不开统计分析，这部分需求在目前企业的分析指标中占了很大的比重，实时的流数据统计可以通过 Flink SQL 方便地实现。而逻辑分析可以进一步细分为风险控制、数据挖掘、用户画像、精准推荐等各个应用场景，如今对实时性要求也越来越高，Flink CEP 就可以作为对流数据进行逻辑分析、进行实时风控和推荐的有力工具。

一、Flume概述

1. Flume可以做什么？

Flume是Cloudera提供的一个高可用的，高可靠的，分布式的海量日志采集、聚合和传输的系统。Flume基于流式架构，灵活简单。分布式不能和集群直接划等号，Flume的分布式不需要搭集群，它的分布式指可以从多个数据源做日志的采集聚合，最终汇聚到一起。

官网：https://flume.apache.org

Flume最主要的作用就是，实时读取服务器本地磁盘的数据，将数据写入到HDFS。所以Flume不是搭在大数据服务器上的，它应该搭在日志服务器上，学习阶段就随便搭了。

下面的爬虫，比如你在京东搜索了华为手机，京东马上给你推荐了华为的其他旗舰手机，这是正常的，那此时你打开淘宝发现淘宝也给你推荐了华为手机，淘宝的数据哪儿来的呢？爬虫爬来的。合理的爬取也是可以的，只要不侵犯别人的正当权益。

2. Flume基础架构

2.1 Agent

Agent是一个JVM进程，它以事件的形式将数据从源头送至目的。Agent主要有3个部分组成，Source、Channel、Sink。

2.2 Source

Source是负责接收数据到Flume Agent的组件。Source组件可以处理各种类型、各种格式的日志数据，常用的source种类有avro、exec、spooling directory、netcat、taildir、kafka。

2.3 Sink

Sink不断地轮询Channel中的事件且批量地移除它们，并将这些事件批量写入到存储或索引系统、或者被发送到另一个Flume Agent。

Sink组件目的地包括hdfs、logger(测试的时候会用，它会把日志打印到控制台)、avro、file roll、HBase、kafka、自定义。

2.4 Channel

Channel是位于Source和Sink之间的缓冲区，它可以存数据，因此Channel允许Source和Sink运作在不同的速率上。Channel是线程安全的，可以同时处理几个Source的写入操作和几个Sink的读取操作。也就是说，在同一个agent里，可以同时存在多个source多个channel多个sink。

Flume自带两种Channel：Memory Channel和File Channel，还有一种kafka channel也会用。

Memory Channel是内存中的队列。Memory Channel在不需要关心数据丢失的情景下适用。如果需要关心数据丢失，那么Memory Channel就不应该使用，因为程序死亡、机器宕机或者重启都会导致数据丢失。

File Channel将所有事件写到磁盘。因此在程序关闭或机器宕机的情况下不会丢失数据。

2.5 Event

传输单元，Flume数据传输的基本单元，以Event的形式将数据从源头送至目的地。Event由Header和Body两部分组成，Header用来存放该event的一些属性，为K-V结构，header通常情况下是空的，一般是用户自己往里写的，Body用来存放该条数据，形式为字节数组。数据在整个Flume的传输过程中都是以event为单位，但是到了HDFS就是真正的数据了。

channel选择器、sink组和拦截器后面再说。

三、Flume进阶

1. Flume事务

第一次讲事务是在MySQL里，事务主要用来保证数据的一致性，事务的四个特性：原子性、一致性、隔离性、持久性。原子性是说操作不可再分，要么都成功要么都失败，一致性是说数据的同步要保证一致，隔离性是说多个事务之间是互相隔离的，持久性是说数据要持久性存储。

Flume为什么有事务呢？它的内部有两个数据传输过程，一个是从source到channel，一个是从channel到sink，Flume中的事务就是保证数据在这两个过程中传输时不会出现问题。比如当source往channel送数据的时候，channel已经满了，数据就可能丢失，当sink从channel拿完数据后，并没有成功sink到下游(比如hdfs)，这时也可能会出现数据丢失，所以Flume有两个事务，put事务和take事务。

source会主动把数据put进channel，sink会主动从channel中拉取数据。

Put事务流程

  • doPut：将批数据先写入临时缓冲区putList，攒够一波后推送给channel，这个putList是每个事物都独有的。(source不是一条一条推送的，它拿到数据后不会马上给channel，而是放到缓冲区batch，这个batch本质上就是个list集合，攒够一波后(一波有多少个event是我们自己配置的transactionCapacity，这个就是batch的容量)再推送给channel)
  • doCommit：尝试提交数据，检查channe内存队列是否足够合并，如果空间足够就提交成功，不够就回滚
  • doRolback：channe内存队列空间不足，回滚数据，它这个回滚不是把数据放回source，而是直接销毁，同时给source报个异常，告诉source这批数据没有提交成功，需要重新采集

Take事务

  • doTake：将数据取到临时缓冲区takeList，并将数据发送到HDFS。(sink也是主动从channel拉取数据，也是有个batch叫takeList，攒够一波后尝试发送给下游)
  • doCommit：如果数据全部发送成功，则清除临时缓冲区takeList，并且channel中对应的数据也清空，发送失败的话回滚
  • doRollback：数据发送过程中如果出现异常，rollback将临时缓冲区takeList中的数据归还给channel内存队列，只要出现了问题整个批次全部回滚
  • 所以flume可能造成数据的重复，比如拉取了100个event到缓冲区，然后sink到下游，sink到第50个event的时候出现问题，这时就会回滚，把100个event全还给channel，sink再次拉取的时候不就有50条是重复的了
  • sink不是每次都能拉到数据的，比如channel里的数据拉完了，新的还没过来，这时就拉不到数据了

这里有个疑问是，sink拉取一批数据，把这批数据放到缓冲区后，channel里还有这批数据吗？

后面自定义source、sink的时候，就会涉及到事务了，从代码的角度看下就知道怎么回事了。

所以Flume可能丢数据吗？

如果用的memory channel是可能丢数据的，file channel基本不可能丢数据，但是可能会出现重复数据。

2. Flume Agent内部原理

event：event是在source里封装的

拦截器：拦截器可以写也可以不写，也可以写多个构成拦截器链

channel选择器：channel选择器有两种，Replicating Channel Selector(复制，默认)和Multiplexing Channel Selector(多路复用)。Replicating会将source过来的events发往所有channel，而Multiplexing可以通过配置headers来决定发往哪些Channel。headers中的配置一般是在拦截器中加，或者是自定义source时在包装event时加，不过如果用的是Flume提供好的source就只能在拦截器中加了，否则要改源代码。

sink组：过了channel后是sink组，sink组来决定每个channel中的event要给到哪些sink，以及怎么给，因为1个sink只能对接1个channel，而1个channel是可以挂多个sink的。SinkProcessor有三种：DefaultSinkProcessor、LoadBalancingSinkProcessor、FailoverSinkProcessor，每种都有其各自的功能。DefaultSinkProcessor对应的是单个的Sink；LoadBalancingSinkProcessor和FailoverSinkProcessor对应的是Sink Group，LoadBalancingSinkProcessor可以实现负载均衡的功能，sink拉取数据时有轮询和随机两种方式；FailoverSinkProcessor有故障恢复的功能，同组同时只能有1个sink在拉取，其他等着，它挂了别的sink再顶上。

一个event在flume内部的经历：

一条Log –> source – event(出生) –> channelprocessor –> interceptors –> channelSelector –> channel –> sinkprocessor –> sink –> event

3. Flume拓扑结构

说是拓扑结构，实际上就是多个agent之间的互相搭配，根据需要爱怎么搭怎么搭，这里只是提供了几种常见搭配。

简单串联

这种模式是将多个flume顺序连接起来了，从最初的source开始到最终sink传送的目的存储系统。此模式不建议桥接过多的flume数量， flume数量过多不仅会影响传输速率，而且一旦传输过程中某个节点flume宕机，会影响整个传输系统。flume提供了专门在多个agent之间传输数据的sink和source，即avro，内部实现就是用的netcat socket？

复制和多路复用

就是把数据通过多个不同的sink发往多个下游。Flume支持将事件流向一个或者多个目的地。这种模式可以将相同数据复制到多个channel中，或者将不同数据分发到不同的channel中，sink可以选择传送到不同的目的地。

负载均衡和故障转移

Flume支持使用将多个sink逻辑上分到一个sink组，sink组配合不同的SinkProcessor可以实现负载均衡和故障转移的功能。

聚合

就是把多个Flume采集到的日志先聚集到1个Flume上，然后再发送到HDFS。这种模式是我们最常见的，也非常实用，日常web应用通常分布在上百个服务器，大者甚至上千个、上万个服务器。产生的日志，处理起来也非常麻烦。用flume的这种组合方式能很好的解决这一问题，每台服务器部署一个flume采集日志，传送到一个集中收集日志的flume，再由此flume上传到hdfs、hive、hbase等，进行日志分析。

4.6 自定义Sink

Sink不断地轮询Channel中的事件且批量地移除它们，并将这些事件批量写入到存储或索引系统、或者被发送到另一个Flume Agent。

Sink是完全事务性的。在从Channel批量删除数据之前，每个Sink用Channel启动一个事务(自定义sink的代码里可见)。批量事件一旦成功写出到存储系统或下一个Flume Agent，Sink就利用Channel提交事务。事务一旦被提交，该Channel从自己的内部缓冲区删除事件。

4. Flume参数调优

Source

增加Source个（使用Tair Dir Source时可增加FileGroups个数）可以增大Source的读取数据的能力。例如：当某一个目录产生的文件过多时需要将这个文件目录拆分成多个文件目录，同时配置好多个Source 以保证Source有足够的能力获取到新产生的数据。

batchSize参数决定Source一次批量运输到Channel的event条数，适当调大这个参数可以提高Source搬运Event到Channel时的性能。

有3个大小需要注意，一是source一次put给channel的event条数，二是channel的事务条数，三是channel的最大容量条数，它们仨的大小关系应该依次递增。4个，还有个sink一次take的event条数。

Channel

type 选择memory时Channel的性能最好，但是如果Flume进程意外挂掉可能会丢失数据。type选择file时Channel的容错性更好，但是性能上会比memory channel差。

使用file Channel时dataDirs配置多个不同盘下的目录可以提高性能。

Capacity 参数决定Channel可容纳最大的event条数。transactionCapacity 参数决定每次Source往channel里面写的最大event条数和每次Sink从channel里面读的最大event条数。transactionCapacity需要大于Source和Sink的batchSize参数。

Sink

增加Sink的个数可以增加Sink消费event的能力。Sink也不是越多越好够用就行，过多的Sink会占用系统资源，造成系统资源不必要的浪费。

batchSize参数决定Sink一次批量从Channel读取的event条数，适当调大这个参数可以提高Sink从Channel搬出event的性能。

5. Flume的事务机制

Flume的事务机制（类似数据库的事务机制）：Flume使用两个独立的事务分别负责从Soucrce到Channel，以及从Channel到Sink的事件传递。比如spooling directory source 为文件的每一行创建一个事件，一旦事务中所有的事件全部传递到Channel且提交成功，那么Soucrce就将该文件标记为完成。同理，事务以类似的方式处理从Channel到Sink的传递过程，如果因为某种原因使得事件无法记录，那么事务将会回滚。且所有的事件都会保持到Channel中，等待重新传递。

6. Flume采集数据会丢失吗?

根据Flume的架构原理，Flume是不可能丢失数据的，其内部有完善的事务机制，Source到Channel是事务性的，Channel到Sink是事务性的，因此这两个环节不会出现数据的丢失，唯一可能丢失数据的情况是Channel采用memoryChannel，agent宕机导致数据丢失，或者Channel存储数据已满，导致Source不再写入，未写入的数据丢失。

Flume不会丢失数据，但是有可能造成数据的重复，例如数据已经成功由Sink发出，但是没有接收到响应，Sink会再次发送数据，此时可能会导致数据的重复。

Hbase

HBase是Hadoop项目中的数据库。它用于需要对大量的数据进行随机、实时的读写操作的场景中。HBase的目标就是处理数据量非常庞大的表，可以用普通的计算机处理超过10亿行数据，还可处理有数百万列元素的数据表。

一、HBase的产生

1. 谷歌的三篇论文

谷歌的三篇论文恰好对应了大数据体系的3个模块

  • GFS，goole file system 2003年发布，HDFS基于它
  • MapReduce 2004年发布，一个编程模型，用来处理和产生大规模数据集
  • BigTable 2006年发布，存储结构化数据，构建于GFS之上，它不是一个关系型数据库，不支持类SQL查询，可以通过键值访问，Hbase基于它，表的数据在GFS当中，表的管理比如元数据什么的在BigTable当中

小知识

  • java与.net：java是2001年开始流行的，它可以在网页里嵌入一些东西，所以之后游戏就流行了，之后c语言模仿java做了.net，也很火，后来分布式出来后，.net就不适用了。

  • 事务：把多个对数据库的操作当成一个整体，要么同时成功要么同时失败，一般说的事务是指单点事务，多个数据库之间的事务叫分布式事务。

  • MySQL一张表的最大列数是4096

为什么有了MySQL还需要hbase呢？

2. MySQL的问题

2.1 查询慢

查询列太多，会影响性能

当表的列很多的情况下，一般称之为宽表，会影响查询性能，一般会 采用垂直拆分的方式将表进行切分

数据行太多，会影响性能

数据文件太大，MySQL单表是有极限值：4G
数据行数一般在500万到2000万之间，会导致数据库阻塞

当表的行很多的情况下，一般称之为高表，会影响查询性能，一般会采用水平拆分的方式将表进行切分，一般会存在切分的规则

表随着业务发生变化，增加新的列(动态列)

比如加个微信号的列，MySQL的话，会给没有微信号的id，也分配内存空间

char和varchar的区别：char会用空格补到指定长度，varchar可变长，为什么一般是varchar(255)呢？MySQL会用1个字节来保存数据的真实长度，如果大于255它会用两个字节来保存数据的真实长度。比如varchar(3)，存了个ab，MySQL会用1个字节来保存数据的真实长度2。早期的MySQL只能当关系型数据库用，8.0之后可以当nosql数据库用，因为它支持了新的数据类型：JSON。

动态列：就是类似JSON的列，有数据就分配空间，没数据就不分配空间

2.2 插入慢

查询慢，可以通过上面的拆表来解决，那么拆表后插入慢该怎么办呢？把一行数据拆成了两张表，那么当一条数据过来要往表里插入的时候，就有个事务的问题。可以先把数据插入内存+顺写日志的方式解决插入慢的问题。

现在MySQL查询慢和插入慢的问题都解决了，那为什么还要有hbase呢？

2.3 MySQL的极限值

MySQL本身的存储有极限值

  • myism : 65535T

  • innodb : 64T，现在都用这个引擎

64T远远不满足大数据的需求，所以才有了Hbase，Hbase就是根据上面说的方式存储数据的。

二、HBase简介

1. 什么是hbase

官网：http://hbase.apache.org

Apache HBase是一个分布式，可伸缩，存储海量数据的Hadoop数据库，注意它基于hadoop。

HBase受到了Google的BigTable论文思想的启发，目前作为Hadoop的子项目来开发维护，用于支持非结构化的数据存储。

从功能上看，HBase是一个高可靠性、高性能、面向列、可伸缩的分布式存储系统，利用HBase技术可在廉价PC Server上搭建起大规模结构化存储集群。HBase的目标是存储并处理大型的数据，更具体来说是仅需使用普通的硬件配置，就能够处理由成千上万的行和列所组成的大型数据。

  • 行式存储：查询效率高，它能根据主键快速定位数据，但是统计效率低
  • 列式存储：查询效率低，统计效率高，所以HBase不是让你快速查询的。hbase的列存储插入快，所以所谓的修改其实也是插入，将旧的数据遮挡住，查询的时候只能查询最新的数据，就好像修改了一样。删除数据其实也可以以插入的方式进行操作，将旧的数据遮挡住，查询的时候无法查询到，并且它是通过时间戳实现遮挡的，直到compact的时候，才会真正删除
  • 列式存储的查询也降低了网络IO，加载表时不会把全部字段加载过来

关于hbase增删改查的效率

增加：使用put指令，时间戳是由Hbase自动生成的，而且存储的是数据的单元格信息，而不是一行数据信息。Hbase本身就是用于存储海量数据的数据库，所以对插入操作进行了优化，性能非常的快，而且Hbase本身就是基于HDFS文件系统，所以数据量增加不是很大的问题。数据插入后，读取数据时，其实依赖于时间戳，hbase会把最新版本的数据查询出来。

修改：修改数据时，逻辑上需要先查询数据，再进行修改，对于Hbase来讲，性能非常低，所以不能采用这种方式来实现修改操作。Hbase采用了新增数据的方式来覆盖旧的数据，达到修改的目的。

删除：Hbase中的删除，其实也是增加新的数据，只不过进行了特殊处理，不让hbase可以查询出来，在合并的时候会真正删除

查询：全表扫描或范围查询用scan，查询单条数据用get

hbase只说了海量数据的存储，可没查询什么事儿。存储就是插入，它的插入很快，但是查询比较慢。修改一般来说也是先查到目标数据然后再修改，但是hbase不查，直接通过插入的方式实现修改

如果要把张三的年龄由30改成40，需要把(张三，30)这条数据查出来，然后再改吗？不需要，HBase一查就慢了，直接用(张三，40)进行插入覆盖，那怎么避免把旧的数据查出来呢？加个时间戳。

2. hbase和mysql的区别

HBase作为一NoSQL数据库，不支持完整的事务性，而且仅仅支持基于RowKey的索引，在性能上不如memcached和redis。但是在海量数据，持久化存储方便比内存类型的NoSQL强的多，作为文档型NoSQL在分布式存储上比mongoDB做切分和MapReduce分析也简单方便的多。这一切都源于HBase本身基于Hadoop，可以简单的通过增加廉价节点的方式进行扩展，对于数据本身就可以很好的进行水平切分，同时和HDFS，MapReduce，Spark等结合的很好。不仅可以方便的进行存储同时可以更加方便的对数据进行处理和运算，这才是HBase最核心的特性。这些都是常见的关系型数据库无法比拟的。比其他常见的NoSQL也要强出不少。

当然，HBase并不能解决所有的问题，所以才会有那么多的NoSQL和SQL，HBase典型的应用场景就是不断的插入新的信息。对于持续的大量的插入可以达到每秒百万的吞吐量。对于已有的数据修改的频率是很小的。Google用自己的BigTable存储互联网上新生成的网页，Facebook的messenger也是基于HBase实现的。

3. hbase的数据结构和概念

nosql数据库基本都是存键值对，后面学的redis也是键值对，所以看起来像一个多维的map

row key相当于主键，用来定位数据，它就是kv键值对的k，水平切分后的数据叫region，类似于hive里的分区，region垂直切分后的每个列族叫store，真正存储时是以列族的方式来存

逻辑上，HBase的数据模型同关系型数据库很类似，数据存储在一张表中，有行有列。但从HBase的底层物理存储结构（K-V）来看，HBase更像是一个multi-dimensional map(多维映射的map)。

region server，分区服务器，布置在几个节点上就有几个region server

master与region server不直接关联，而是通过zookeeper进行关联，region server主要是对数据做操作，master主要是对表做操作。所以一定要先启动zk后启动hbase

region体现了水平拆分

hbase是以kv对的形式存储的，rowkey相当于k，后面的数据相当于v

rowkey不能太简单也不能太复杂，设计rowkey是个技术活

Namespace：命名空间，类似于关系型数据库的database概念，每个命名空间下有多个表。HBase有两个自带的命名空间，分别是hbase和default，hbase中存放的是HBase内置的表，刚开始有两张，meta表和namespace表，default表示用户默认使用的命名空间。

Table：类似于关系型数据库的表概念。不同的是，HBase定义表时只需要声明列族即可，不需要声明具体的列，所以又有动态列的说法。但是一定要声明列族，非结构化数据库，可以没有某些列，所以列族是表的一部分，列不是。所以对于hbase来说，难点在于表该怎么去设计。

Rowkey：行键，类似于关系型数据的主键，所以不能重复，HBase表中的每行数据都由一个RowKey和多个Column（列）组成，数据是按照RowKey的字典顺序升序存储的，并且查询数据时只能根据RowKey进行检索，所以RowKey的设计十分重要。

Column：动态列，HBase中的每个列都由列族和列名进行限定，例如info：name，info：age。建表时，只需指明列族，而列限定符无需预先定义。不同列族里面可以有相同的列名，列族体现了垂直拆分。

Region：类似于分区的概念，需要考虑数据的水平切分，相同key的数据一定在同一个region

ColumnFamily：列族是表结构的一部分，所以创建表时列族应该是确定的，但是列是不确定的，随着业务的发展，可能会增加列。列族可以通过修改表结构的方式添加，不可以动态添加。

Store：存储，Hbase将一个列族的数据存储在一起，一个表存在多个列族，那么就会存储在不同的store里，底层都是以字节码存的

Timestamp：用于标识数据的不同版本（version），每条数据写入时，系统会自动为其加上该字段，其值为写入HBase的时间。被删除的数据只是被隐藏了，在合并的时候才会被真正删除。

  • hbase中的版本号就是类似于加锁的概念，是一种乐观锁，它不通过事务来完成，有事务就慢了

  • 比如有条数据(1, lisi, 40, v1)。现在两个人同时对这条数据做更改，各改各的，A同学改完放回去，数据变成了(1, lisi, 30, v2)。B同学改完放回去的时候发现，他取数据的时候版本号是v1，放回去时版本号已经变了，说明他当时拿的数据就不是最新的了，于是修改失败，所以这样就可以不通过事务实现加锁

Cell：由{rowkey, column Family：column Qualifier, time Stamp} 唯一确定的单元。cell中的数据是没有类型的，全部是字节码形式存储，都是字节码便于压缩。

1. HBase架构

2. HBase核心组件和概念

HMaster：Master是所有Region Server的管理者

  • 对于表的操作：create, delete, alter
  • 对于RegionServer的操作：分配regions到每个RegionServer(就是决定Region放哪个RegionServer里)，监控每个RegionServer的状态，负载均衡和故障转移
  • 原则上HMaster挂了后集群还能正常运行一段时间，因为只查数据的话HRegionServer就够了，直到发生对表的操作集群才会崩溃。那么插入数据行不行呢？不一定行，比如刚好该切分region的时候，如果再插入数据，就涉及到对region的切分，这个时候就要用到master了

HRegionServer：是一个java进程，其实现类为HRegionServer，Hbase中用于管理内部存放的Region

  • 对于数据的操作：get, put, delete
  • 对于Region的操作：splitRegion, compactRegion
  • 里面可以有多个region，多个store，多个WAL和1个读缓存
  • 进程都是有main方法的，关于进程和线程的区别？

HRegion：类似于水平拆分的表（表的一段数据），类似于Kafka中的分区的概念，分区的rowkey是有序的。如果一张表数据量比较小，那一张表就是一个region

HStore：用于管理Table的列族，每个store对应1个列族，包括写缓存和storefile，一个Region中可能有多个Store

HStoreFile：保存实际数据的物理文件，StoreFile以Hfile的形式存储在HDFS上，是个有序的K-V的文件。每个Store会有一个或多个StoreFile，数据在每个StoreFile中都是有序的。

BlockCache：读缓存，每次新查询的数据会缓存在BlockCache中，用于提高查询效率

MemStore：写缓存，由于HFile中的数据要求是有序的，所以数据是先存储在MemStore中，排好序后，等到达刷写时机才会刷写到HFile(也是为了memory中数据太多占用过多内存)，每次刷写都会形成一个新的HFile。另一方面也是为了提高效率，插入数据时，如果直接和Hbase底层文件交互，那么效率比较低，所以可以将数据直接写入到内存中然后返回，等攒够一批数据再刷写。MemStore有溢写功能，如果溢写太频繁性能也会降低，所以有个优化点就是提高写缓存的容量。

WAL：Write Ahead Log，预写日志，由于数据要经MemStore排序后才能刷写到HFile，但把数据保存在内存中会有很高的概率导致数据丢失，为了解决这个问题，数据会先写在一个日志文件中，然后再写入MemStore中。所以在系统出现故障的时候，数据可以通过这个日志文件重建。课件的图上是每个regionserver有1个WAL，实际是不对的，应该是每个region有1个WAL。刷写的时候，把memstore里的数据刷写到磁盘，然后通过事务编号去预写日志里删除对应的日志数据。

Zookeeper：HBase通过Zookeeper来做master的高可用、RegionServer的监控、元数据的入口以及集群配置的维护等工作。连接就是在连接zookeeper。HBase默认有内置的ZK，但是一般不用

HDFS：HDFS为Hbase提供最终的底层数据存储服务，同时为HBase提供高可用的支持。hbase基于hdfs，数据最终在hdfs上存着。

hbase会丢数据吗？一般不会，除了写缓存里的数据，hbase只是个应用程序，它不存数据，数据在hadoop上存着，所以hbase不管怎么宕机都不会丢数据，重启就完事儿了

3. HBase写流程

写流程：

  • 为了能够确定增加数据时，数据在哪一个机器的region中，所以首先客户端查询hbase：meta表信息，但是由于meta也是一张表，这张表在哪里不确定。所以必须先访问zookeeper，获取meta的表位置。Client先访问zookeeper，目的是获取要访问的表在哪个region server的region里，hbase:meta表位于哪个Region Server(可以用scan ‘hbase:meta’看看这个表里都有啥，也可以在zookeeper里通过get /hbase/meta-region-server查看meta表在哪个regionserver里)
  • 访问对应的Region Server，获取hbase:meta表，根据读请求的namespace:table/rowkey，查询出目标数据位于哪个Region Server中的哪个Region中。并将该table的region信息以及meta表的位置信息缓存在客户端的meta cache，方便下次访问
  • 与目标Region Server进行通讯，将数据顺序写入（追加）到WAL
  • 将数据写入对应的MemStore，数据会在MemStore进行排序
  • 向客户端发送ack
  • 等达到MemStore的刷写时机后，将数据刷写到HStoreFile

问题1：为什么MySQL找表不需要先访问zookeeper？meta表的数据如果放在3台机器上，那么zookeeper怎么知道该访问的meta表位于哪个regionserver上呢？

MySQL是单点的，一般分布式的才需要去找zookeeper，因为同张表的数据可能在3个不同的机器上。而meta表只会有1个region，不可能位于多台机器上。

问题2：为什么要有写缓存和WAL？

为了写的快，直接跟底层文件进行交互很慢，所以写到缓存，但是又怕缓存里的数据丢失，所以先写到WAL，往WAL写是顺写，很快。

4. Hbase读流程

  • Client先访问zookeeper，获取hbase:meta表位于哪个Region Server
  • 访问对应的Region Server，获取hbase:meta表，根据读请求的namespace:table/rowkey，查询出目标数据位于哪个Region Server中的哪个Region中。并将该table的region信息以及meta表的位置信息缓存在客户端的meta cache，方便下次访问
  • 与目标Region Server进行通讯，分别在读缓存、写缓存和Store File（HFile）中查询目标数据，并将查到的所有数据进行合并(此处所有数据是指同一条数据的不同版本（time stamp）或者不同的类型（Put/Delete）)，但是有HStoreFile可能很多，所以为了避免查询效率低，需要对文件进行过滤，先采用布隆过滤器进行过滤，然后使用rowkey和时间对文件进行筛选，最终找到我们需要的文件。
  • 将查询到的新的数据块（Block，HFile数据存储单元，默认大小为64KB）缓存到Block Cache
  • 将合并后的最终结果返回给客户端

读为什么比写慢，因为它读的地方多，先在读缓存里读，没有再往写缓存里读(因为刚刚写的东西被读取的概率大)，没有再去storefile里读，读到的话往读缓存里放一份然后返回，怕的是读的时候，有部分数据在读缓存里，有部分数据在写缓存里，有部分数据在storefile里，这样读到后还需要merge合并。从文件中读数据的时候，也不会把这个region里的数据全读，而是先通过时间范围、rowkey范围、布隆过滤器等判断肯定不在哪个文件里，然后再去可能在的文件里读。这样整个流程下来，读的速度就慢了。所以hbase设计的初衷不是为了快速查询，而是快速写入。

5. 布隆过滤器

原理跟hashmap很像，zhangsan在不在里面我不知道，但是他不在里面我肯定知道，hbase读文件的时候就用到了这个原理。

布隆过滤器只能校验数据不存在我们的环境中，可以设置多级过滤。比如zhangsan第一次访问过我们的网站，hash后被分到了一级过滤的1号位，发现是个白色，代表没人来过，我们就知道”zhangsan以前从来没访问过”，然后把1号位标记成蓝色，当zhangsan再访问的时候，hash一下又到了1号位，发现是蓝色，于是我们知道”zhangsan可能访问过”，然后”wangwu”来了，hash后也被分到了1号位，于是我们知道”wangwu可能访问过”，继续hash走二级过滤，发现是白色，于是我们知道”wangwu以前从来没访问过”，再把二级过滤的1号位标记成红色，代表有人访问过。注意这里记录的不是个人信息，而是这个位置也没有人访问，存个人信息的话数据量就太大了。

redis的位图bitmap也跟布隆过滤器有关，可以用0和1标记这个位置也没有人访问过，这样1个字节能保存8个用户的访问状态，1M就能保存8*1024*1024=8千万个用户的访问状态。

6. flush刷写

关于flush，不管是达到阈值还是人工刷写，只要执行一次flush，hdfs上就会生成1个文件，哪怕只有1条数据

再执行major_compact ‘test:emp’压缩后，会生成第3个合并后的文件，过一段时间后，前2个小文件会自动删除

如果合并后的文件过大，还会涉及到切分，刷写、合并和切分会在以下三节来讲，而且这三个过程的实现方式比较复杂，大概了解下流程即可。

MemStore刷写时机：

  • 当Region中所有的memstore的大小超过了hbase.hregion.memstore.flush.size(默认值128M)，其所在region的可能所有memstore都会刷写，或者选择部分的memstore进行刷写。当Region中所有的memstore的大小达到了128M * 4时，会阻止继续往该memstore写数据
  • 当region server中memstore的总大小达到 java_heapsize * 0.4 * 0.95，region会按照其所有memstore的大小顺序(由大到小)依次进行刷写。直到region server 中所有memstore的总大小减小到上述值以下。当region server中memstore的总大小达到java_heapsize * 0.4时，会阻止继续往所有的memstore写数据

  • 到达自动刷写的时间，也会触发memstore flush。自动刷新的时间间隔由该属性进行配置hbase.regionserver.optionalcacheflushinterval(默认1小时)

  • 当WAL文件的数量超过hbase.regionserver.max.logs，region会按照时间顺序依次进行刷写，直到WAL文件数量减小到hbase.regionserver.max.log以下 (该属性名已经废弃，现无需手动设置，最大值为32)

7. Compaction合并

由于memstore每次刷写都会生成一个新的HFile，且同一个字段的不同版本（timestamp）和不同类型（Put/Delete）有可能会分布在不同的HFile中，因此查询时需要遍历所有的HFile。为了减少HFile的个数，以及清理掉过期和删除的数据，会进行StoreFile Compaction。

Compaction分为两种，分别是Minor Compaction和Major Compaction。Minor Compaction会将临近的若干个较小的HFile合并成一个较大的HFile，并清理掉部分过期数据(实际不怎么清理)。Major Compaction会将一个Store下的所有的HFile合并成一个大HFile，并且会清理掉所有过期和删除的数据。最少要有3个hfile才能触发Major Compaction。

生产环境中，大合并是要禁用的，太影响性能，由人工来做。

8. split切分

默认情况下，每个Table起初只有一个Region，随着数据的不断写入，Region会自动进行拆分。刚拆分时，两个子Region都位于当前的Region Server，但处于负载均衡的考虑，HMaster有可能会将某个Region转移给其他的Region Server。

Region Split时机：

具体的切分策略为：initialSize*tableRegionsCount*tableRegionsCount*tableRegionsCount，initialSize = 256M

第一次split：1^3 * 256 = 256MB ，第二次split：2^3 * 256 = 2048MB ，第三次split：3^3 * 256 = 6912MB ，第四次split：4^3 * 256 = 16384MB > 10GB，因此取较小的值10GB ，后面每次split的size都是10GB了。

Hbase 2.0引入了新的split策略：如果当前RegionServer上该表只有一个Region，按照2 * hbase.hregion.memstore.flush.size分裂，否则按照hbase.hregion.max.filesize分裂。

疑问：Hbase的region划分，每次划分大小都一样吗

有可能不一样：因为hbase要保证同一个rowkey的数据要在一个store中

六、HBase优化

1. 预分区

每一个region维护着startRow与endRowKey，如果加入的数据符合某个region维护的rowKey范围，则该数据交给这个region维护。那么依照这个原则，我们可以将数据所要投放的分区提前规划好，以提高HBase性能。默认情况下，一个Table只有一个Region，所有的rowkey都应该放置在一个region里，只有一个region的话，它的默认rowkey的范围是 -∞, +∞。如果用默认的分区规则容易出现热点分区，比如第一次切分后有两个分区(负无穷,50)，(50,正无穷)，如果原数据用了自增键，后面过来的数据rowkey可能都大于50，数据就都会往第二个分区放，第二个分区就是热点分区。而且不做预分区的话，数据量不断增大，就会不断发生合并切分，也影响效率。所以一般进行分区规划时，以10G为单位，对数据进行评估。

注意：同一个rowkey的数据不能放置在不同的region中

2. RowKey设计

一条数据的唯一标识就是rowkey，类似于关系型数据库的主键。在HBase中，rowkey是有序的。由于HBase存在分区的概念，所以当存储数据时，这条数据存储于哪个分区，就取决于rowkey处于哪一个预分区的区间内。设计rowkey的主要目的 ，就是让数据均匀的分布于所有的region中，在一定程度上防止数据倾斜。接下来我们就谈一谈rowkey常用的设计方案。

rowkey的设计原则：唯一原则、长度原则、散列原则。

  • 唯一性原则
  • 长度原则，rowkey设计时不要太长，10~100Bytes
  • 散列原则，为了避免数据倾斜的问题，可以让rowkey分散一下放置在不同的region中
  • 归纳原则，为了查询的方便，一般会将有规律的rowkey数据存储在一起

生成随机数、hash、散列值

在做此操作之前，一般我们会选择从数据集中抽取样本，来决定什么样的rowKey来Hash后作为每个分区的临界值。注意这样就没法取范围数据了

字符串反转

字符串拼接，又叫加盐，在原数据后面加随机数

3. 内存优化

HBase操作过程中需要大量的内存开销，毕竟Table是可以缓存在内存中的，但是不建议分配非常大的堆内存，因为GC过程持续太久会导致RegionServer处于长期不可用状态，一般16~36G内存就可以了，如果因为框架占用内存过高导致系统内存不足，框架一样会被系统服务拖死。

七、HBase与Hive的对比

1. HBase与Hive的对比

Hive

  • 数据仓库：Hive的本质其实就相当于将HDFS中已经存储的文件在Mysql中做了一个双射关系，以方便使用HQL去管理查询
  • 用于数据分析、清洗：Hive适用于离线的数据分析和清洗，延迟较高
  • 基于HDFS、MapReduce：Hive存储的数据依旧在DataNode上，编写的HQL语句终将是转换为MapReduce代码执行

HBase

  • 数据库：是一种面向列族存储的非关系型数据库
  • 用于存储结构化和非结构化的数据：适用于单表非关系型数据的存储，不适合做关联查询，类似JOIN等操作
  • 基于HDFS：数据持久化存储的体现形式是HFile，存放于DataNode中，被ResionServer以region的形式进行管理
  • 延迟较低，接入在线业务使用：面对大量的企业数据，HBase可以直线单表大量数据的存储，同时提供了高效的数据访问速度

3. 几个面试题

1. hbase热点问题

Hbase热点(数据倾斜)问题，读写请求会集中到某一个RegionServer上，如何处理？

产生问题的原因

  • Hbase中rowkey是按照字符串的字典顺序排列和存储的。所以当大量连续的rowkey数据集中写在个别的的region中，导致region数据不平衡
  • 创建表时，没有预分区。只有一个region
  • 存在预分区，设计的rowkey没有打散分布，可能放置在一个region

解决方案

  • Rowkey设计时，尽量不要采用时间戳，最好使用随机数+业务主键
  • Rowkey设计时不要太长(10-100Byte)，太长的话会占内存，因为每条数据前面都有rowkey
  • Rowkey设计时将同一批(查询)数据保存在一起，比如把2022年的数据放一起，其中又把每个月的数据放一起
  • 设计预分区

4. 几个概念

去中心化

redis有个去中心化的概念，比如hbase，访问hbase时一般不是访问hbase本身，而是访问zookeeper，通过zookeeper去找hbase，hbase又通过master去找regionserver，比如kafka，通过leader去找follower，比如spark，通过driver去找executor，这个就是中心化，而redis，你可以访问任意一个节点，访问其中一个就等于访问整个集群，而且数据在整个集群中还只存一份。去中心化是一种特殊的分布式操作，主流的还是主从。

稀疏表和密集表

密集表比如MySQL，如果一个字段为空，实际上它也是占内存的，稀疏表比如hbase，一个列为空就不占内存。

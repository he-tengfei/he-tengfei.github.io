一、HDFS概述

下面7章细说HDFS

1. HDFS的产生背景

随着数据量越来越大，在一个操作系统存不下所有的数据，那么就分配到更多的操作系统管理的盘中，但是不方便管理和维护。比如一份文件拆成了三份放到不同的机器，当我需要查看这个文件的时候怎么把这三份文件拼成一份文件呢？这时就迫切需要一种系统来管理多台机器上的文件，这就是分布式文件管理系统，HDFS只是分布式文件管理系统中的一种。

2. HDFS定义

HDFS（Hadoop Distributed File System），它是一个文件系统，用于存储文件，通过目录树来定位文件；其次，它是分布式的，由很多服务器联合起来实现其功能，集群中的服务器有各自的角色。

HDFS的使用场景：适合一次写入，多次读出的场景，且不支持文件的随机修改。适合用来做数据分析，并不适合用来做网盘应用。这里的随机修改指，想对某个文件的某一行做修改。

3. 优缺点

优点

  • 高容错性
    • 数据自动保存多个副本。它通过增加副本的形式，提高容错性。当某一个副本丢失以后，它可以自动恢复。比如你有5台机器，A文件存了3个副本，你不小心破坏了其中1个副本，NN会检测到A文件只剩下2个副本了，然后就会自动再备份1个副本
  • 适合处理大数据
    • 数据规模：能够处理数据规模达到GB、TB、甚至PB级别的数据
    • 文件规模：能够处理百万规模以上的文件数量，数量相当之大
  • 可构建在廉价机器上，通过多副本机制，提高可靠性

缺点

  • 不适合低延时数据访问，比如毫秒级的存储数据，是做不到的

  • 无法高效的对大量小文件进行存储

    • 存储大量小文件的话，它会占用NameNode大量的内存来存储文件目录和块信息。这样是不可取的，因为NameNode的内存总是有限的。NN存1000个100M的大文件和1000个1M的小文件占用的内存是差不多相同的，但是储存量却差了100倍。元数据不管大小字节基本都是150个字节
    • 整个hadoop都害怕小文件，存的时候怕，算的时候也怕，因为NN是运行在内存里的
    • 小文件存储的寻址时间会超过读取时间，它违反了HDFS的设计目标
    • 小文件就存MySQL就行，HDFS设计的初衷就是大数据

  • 不支持并发写入、文件随机修改

    • 一个文件只能有一个写，不允许多个线程同时写
    • 仅支持数据append（追加），不支持文件的随机修改

4. HDFS组成架构

metadata元数据，rack机架，block块

NameNode：就是Master，它是一个主管、管理者

  • 管理HDFS的名称空间
  • 配置副本策略
  • 管理数据块（Block）映射信息
  • 处理客户端读写请求

DataNode：就是Slaver，NameNode下达命令，DataNode执行实际的操作

  • 存储实际的数据块
  • 执行数据块的读/写操作

Client：就是客户端，严格来说并不是HDFS架构里面的

  • 文件切分，文件上传HDFS的时候，Client将文件切分成一个一个的Block，然后进行上传
  • 与NameNode交互，获取文件的位置信息
  • 与DataNode交互，读取或者写入数据
  • Client提供一些命令来管理HDFS，比如NameNode格式化
  • Client可以通过一些命令来访问HDFS，比如对HDFS增删查改操作
  • 关于客户端的理解，命令行里敲的命令就是客户端启动的，在Linux里敲命令shell是客户端，在java代码里敲命令java就是客户端

Secondary NameNode：并非NameNode的热备。当NameNode挂掉的时候，它并不能马上替换NameNode并提供服务

  • 辅助NameNode，分担其工作量，比如定期合并Fsimage和Edits，并推送给NameNode
  • 在紧急情况下，可辅助恢复NameNode
  • 热备是指，主节点挂了从节点能马上顶上

5. HDFS文件块大小

上传文件时，会真正的把文件切分成块，然后再上传，在3.1.3版本默认是128M，可以修改。同个文件多个块的上传下载都是串行的。

为什么默认是128M？

  • 2的n次方
  • 有人统计过，寻址时间为传输时间的1%为最佳状态，假设寻址时间10ms，传输时间就是1s，普通磁盘传输速度按100M/s算，块大小就是100M，接近128M。但实际固态硬盘的速度已经远远超过100M/s了，生产环境中块大小往往调到256M，甚至512M都是有可能的

为什么块的大小不能设置太小，也不能设置太大？

  • HDFS的块设置太小，会增加寻址时间，程序一直在找块的开始位置

  • 如果块设置的太大，从磁盘传输数据的时间会明显大于定位这个块开始位置所需的时间。导致程序在处理这块数据时，会非常慢

  • 总结：HDFS块的大小设置主要取决于网盘传输速率

四、HDFS的数据流

1. HDFS写数据流程

关于通道pipeline的建立，客户端会从返回的3个节点中选择一个最近的，后面会说。传输数据的时候也不是一次性把128M全传过去，而是以packet为单位，大概64kb。块儿数大于1的话，从3开始循环。

假设有3个副本，NN返回3个节点，写的过程中有个节点挂了怎么办？挂不挂写不说，NN怎么知道这3个副本都写成功了呢？2.7.2的DFSOutputStream类的注释说明了这个问题，一是会有ACK机制，如果只有两个响应了ACK，会把没响应的剔除出去，重新建立管道，少的那个副本NN后面会补上。

2. 网络拓扑-节点距离计算

在HDFS写数据的过程中，NN会选择最近距离的DN接收数据，那么这个最近距离怎么计算呢？节点距离：两个节点到达最近的共同祖先的距离总和。

3. 副本存储节点选择

第一个副本在Client所处的节点上，如果客户端在集群外，随机选一个。第二个副本在另一个机架的随机一个节点，第三个副本在第二个副本所在机架的随机节点。

五、NN和2NN的工作机制

1. NN和2NN的工作机制

NN元数据信息维护到哪里？

对HDFS的数据修改(比如经常需要进行随机访问，还有响应客户请求)就一定会涉及到对元数据的修改，如果考虑数据的可靠性，需要将元数据维护到磁盘，带来的问题是对磁盘的数据修改效率低。如果考虑数据访问和修改的效率，需要将元数据维护到内存(只要对磁盘的修改就要走IO，而IO性能损耗非常严重，直接操作内存就不需要走磁盘了，效率很高)，但是带来的问题是数据不可靠。所以综合考虑就是元数据放在内存，并且产生在磁盘中备份元数据的镜像文件fsimage。

如何保证内存和磁盘的数据同步？

采用磁盘+内存的话，那发生数据更改时，是改内存还是改磁盘呢？如果两个都改就又绕回了第一个问题，如果只改一个，如何保证两者之间的数据同步呢？Hadoop提供了这么一个方案：在内存中维护元数据，且在磁盘中通过镜像文件fsimage来维护元数据，并通过编辑日志文件edits记录对元数据的修改操作，记录的方式为文件末尾追加操作(这个效率是很高的，kafka也有类似的机制)。这样就解决了磁盘修改的效率问题，如果内存挂掉了，可以以镜像文件fsimage为基础，再把日志文件edits里记录的操作做一遍，就恢复到了内存挂掉之前元数据的样子。

edits文件记录的操作越来越多怎么办？

但是又带来了新的问题，edits文件记录的操作越来越多，文件越来越大，一旦断点，恢复数据的时间就会越来越长。这时就需要定时的将fsimage文件和edits文件进行合并，旧fsimage + edits = 新fsimage，旧edits就可以不用了，这样就不会导致edits文件越来越大。

由谁来做合并呢？

让NN来做是非常合适的，毕竟元数据就是它管理的，但是NN工作压力很大，就找个秘书2NN来做。大概的过程就是，当满足要合并的条件后，2NN会将NN中的fsimage和当前正在使用的edits文件拉取过来，将fsimage和edits在2NN的内存中进行合并，合并完后生成一个新的fsimage文件，再将新的fsimage文件推送给NN，替换NN中旧的fsimage文件。

这个机制的目的只有一个，就是保证内存中的元数据 = fsimage.chkpoint + edits_inprogress

第一阶段：NN启动

第一次启动NN格式化后，创建Fsimage和Edits文件。如果不是第一次启动，直接加载编辑日志和镜像文件到内存进行合并，以保证磁盘和内存中的数据同步。当接收客户端对元数据进行增删改的请求后，NN先记录操作日志，然后在内存中对元数据进行增删改。

第二阶段：2NN工作

2NN会定时询问(1分钟问一次)NN是否需要CheckPoint，如果满足触发条件，2NN会请求执行CheckPoint。这时NN会滚动正在写的Edits日志文件，然后将滚动前的编辑日志和镜像文件拷贝到2NN。2NN加载编辑日志和镜像文件到内存，并合并生成新的镜像文件fsimage.chkpoint。然后拷贝fsimage.chkpoint到NN，NN将fsimage.chkpoint重新命名成fsimage，把原来的替换掉。

checkpoint触发条件：定时时间到(1个小时)，或者edits中记录的操作达到100万次。设定两个触发条件的原因是为了防止10个小时就做5次操作或者5分钟就做了100万次操作这种情况。

六、DN的工作机制

1. DN的工作机制

一个数据块在DataNode上以文件形式存储在磁盘上，包括两个文件，一个是数据本身，一个是元数据包括数据块的长度，块数据的校验和，以及时间戳。

DataNode启动后向NameNode注册，通过后，周期性（现版本是6小时，PPT上写错了）的向NameNode上报所有的块信息。

心跳(就是通信)是每3秒一次，心跳的目的是告诉NN自己工作状态正常，并返回结果带有NameNode给该DataNode的命令，如复制块数据到另一台机器，或删除某个数据块。如果超过10分钟没有收到某个DataNode的心跳，则认为该节点不可用。

集群运行中可以安全加入和退出一些机器。

2. 数据完整性

思考：如果电脑磁盘里面存储的数据是控制高铁信号灯的红灯信号（1）和绿灯信号（0），但是存储该数据的磁盘坏了，一直显示是绿灯，是否很危险？同理DataNode节点上的数据损坏了，却没有发现，是否也很危险，那么如何解决呢？

如下是DataNode节点保证数据完整性的方法：

自校验的过程：当DataNode读取Block的时候，它会计算CheckSum，如果计算后的校验和CheckSum，与Block创建时值不一样，说明Block已经损坏，DN会主动把这个损坏的块删掉，每6个小时DN会给NN上报块信息，然后NN就会发现这个文件只剩两个副本，NN就会告诉别的有副本的DN再增加一个副本。DataNode在其文件创建后也会周期验证CheckSum。

读取的过程：发现校验和不一样，Client就读取其他DataNode上的Block。

大意就是说，块创建的时候有个checksum1，块上传后有个checksum2，如果checksum1 = checksum2说明文件没有损坏。

有很多种校验算法，比如crc、md5、sha1等等。

3. 掉线时限参数设置

上面说了3秒钟心跳，比如可能由于网络原因3秒钟没有收到心跳，肯定不能认为相应的DN就挂了，默认10分30秒后还没有收到心跳就认为DN挂了，这段时间成为超时时长timeout，可以设置。

4. 服役新数据节点

随着公司业务的增长，数据量越来越大，原有的数据节点的容量已经不能满足存储数据的需求，需要在原有集群基础上动态添加新的数据节点。一般在大促销比如双十一的时候会进行服务器扩容，之后再退役。

注意改完ip和主机名称，删完data和log文件后，原主机上的东西不会对集群产生影响，比如原来104上有2NN，新105上就不会有，因为104上有2NN是因为2NN的配置文件里配置了104，而不是有了配置文件才有了2NN，这个配置文件每台机器都有。

5. 退役旧数据节点

添加到白名单的主机节点，都允许访问NameNode，不在白名单的主机节点，都会被直接退出(会发生数据丢失)。添加到黑名单的主机节点，不允许访问NameNode，会在数据迁移后退出。实际情况下，白名单用于确定允许访问NameNode的DataNode节点，内容配置一般与workers文件内容一致， 黑名单用于在集群运行过程中退役DataNode节点。这俩名单要在搭建集群的时候就规划好，这里略过，用的时候翻文档。

6. Datanode多目录配置

DataNode也可以配置成多个目录，每个目录存储的数据不一样。即：数据不是副本。

七、Hadoop新特性

1. Hadoop2.x新特性

1.1 小文件存档

解决存储小文件办法之一

HDFS存档文件或HAR文件，是一个更高效的文件存档工具，它将文件存入HDFS块，在减少NameNode内存使用的同时，允许对文件进行透明的访问。具体说来，HAR文件看似是个打包以后的文件，实际上我们是可以看到里面是有多少个小文件的，HDFS存档文件对内还是一个一个独立文件，对NN而言却是一个整体，减少了NN的内存。

下面这个就是打包后的har文件，它是个目录，不管3个还是300个小文件，打包成har文件后生成的都是下面4个东西，对于HDFS来说，整个har文件就是一个整体，是一个文件，我们看的话要通过lsr命令。

解决存储小文件办法之二

减少小文件的生成，比如日志文件一个小时滚动一次，改成一天滚动一次。

1.2 回收站trash

开启回收站功能，可以将删除的文件在不超时的情况下，恢复原数据，起到防止误删除、备份等作用。

通过程序删除的文件不会经过回收站，需要调用moveToTrash()才进入回收站。

1.3 集群间数据拷贝

采用distcp命令实现两个Hadoop集群之间的递归数据复制。

2. Hadoop3.x新特性

纠删码机制：

主要解决了hadoop存储数据时的数据冗余问题，与默认三个副本机制相比， EC策略可以节省约50％左右的存储空间。

但不可忽略的是编解码的运算会消耗CPU资源。纠删码的编解码性能对其在HDFS中的应用起着至关重要的作用，如果不利用硬件方面的优化就很难得到理想的性能。英特尔的智能存储加速库（ISA-L）提供了对纠删码编解码的优化，极大的提升了其性能。

纠删码是hadoop3.x新加入的功能，之前的hdfs都是采用副本方式容错，默认情况下，一个文件有3个副本，可以容忍任意2个副本（datanode）不可用，这样提高了数据的可用性，但也带来了2倍的冗余开销。例如3TB的空间，只能存储1TB的有效数据。而纠删码则可以在同等可用性的情况下，节省更多的空间，以RS-6-3-1024K这种纠删码策略为例子，**6份原始数据，编码后生成3份校验数据，一共9份数据，只要最终有6份数据存在，就可以得到原始数据，它可以容忍任意3份数据不可用(6份原始数据用3副本存的话就要存18份)**。

HDFS篇完。

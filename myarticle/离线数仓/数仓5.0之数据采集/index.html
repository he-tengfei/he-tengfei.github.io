一、数据仓库概念

数据仓库（Data Warehouse），是为企业所有决策制定过程，提供所有系统数据支持的战略集合。通过对数据仓库中数据的分析，可以帮助企业，改进业务流程、控制成本、提高产品质量等。数据仓库，并不是数据的最终目的地，而是为数据最终的目的地做好准备。这些准备包括对数据的：清洗，转义，分类，重组，合并，拆分，统计等等。

二、项目需求及架构设计

业务数据的主要来源是MySQL，需要把MySQL里的数据导入到HDFS，可以用datax或maxwell，用户行为日志通过flume或logstash导入到kafka？爬虫数据一般用python来做，业务数据、用户行为日志、爬虫数据，都会被送入kafka，这是收集，存储用的就是HDFS，做数据分析的计算引擎用hive或spark。

问题一：为什么kafka不能直接传到HDFS呢？还要经过flume？

用了flume就不用自己写代码了，不用flume就得自己写代码，写个HDFS消费者，两种方式选哪个都行

kafka和flume看起来很像，但却是完全不一样的两个框架，结构上的相似只是个错觉，kafka本质上是个消息队列，目的是让数据在这里缓存一下，通过缓存达到削峰解耦的目的，kafka的三个组件，生产者producer，服务器broker，消费者consumer，kafka主要做的是broker，producer和consumer其实只是api，不算kafka真正的组件。flume做的是数据采集，数据采集就会有多种数据来源和多种数据去向，它要考虑的是我要从哪个地方采集，采集到哪个地方去，flume的三个组件source、channel、sink，重心在source和sink，flume通过多种多样的source和sink实现了对多种数据源和数据去向的高兼容性。另外kafka是分布式的，而flume是个单进程。这里用flume就是图个省事儿，不用手写kafka的hdfs消费者代码了，有flume就不需要多造轮子解决需求问题了。

问题二：为什么要加个kafka，直接用flume把logFile采集到HDFS不行吗？

缓冲削峰？flume的channel也有缓冲削峰的作用。用kafka确实是能提高性能，它读写比flume快(单节点filechannel峰值大概二三十兆每秒，kafka算上压缩的话可以达两三百兆每秒)，但是代价是集群资源上升了，而且logFile是个文件，死文件，把它上传到hdfs根本不需要做削峰，单台flume一天可以写180G，三台就是500G，也能满足需求。在这里加kafka也不是为了高并发，而是为了给Flink做实时用的，既然离线和实时都需要用到这些数据，那我们就把这些数据放到kafka里做缓存好了，不用再单独写个数据采集的通道了

小知识：一天是86400秒，datax比sqoop的性能要好些

服务器选择物理机还是云主机？

物理机：

  • 以128G内存，20核物理CPU，40线程，8THDD和2TSSD硬盘，戴尔品牌单台报价4W出头。一般物理机寿命5年左右

  • 以阿里云为例，差不多相同配置，每年5W
  • 很多运维工作都由阿里云完成，运维相对较轻松

企业选择：

  • 金融有钱公司和阿里没有直接冲突的公司选择阿里云
  • 中小公司、为了融资上市，选择阿里云，拉倒融资后买物理机
  • 有长期打算，资金比较足，选择物理机

2.5 集群资源规划设计

如何确认集群规模？（假设：每台服务器8T磁盘，128G内存）

（1）每天日活跃用户100万，每人一天平均100条：100万*100条=1亿条

（2）每条日志1K左右，每天1亿条：100000000/1024/1024=约100G

（3）半年内不扩容服务器来算：100G*180天=约18T

（4）保存3副本：18T*3=54T

（5）预留20%-30%Buf=54T/0.7=77T

2. 数据埋点

2.1 主流埋点方式（了解）

目前主流的埋点方式，有代码埋点（前端/后端）、可视化埋点、全埋点三种。埋点这活一般不会让大数据开发参与。

代码埋点是通过调用埋点SDK函数，在需要埋点的业务逻辑功能位置调用接口，上报埋点数据。例如，我们对页面中的某个按钮埋点后，当这个按钮被点击时，可以在这个按钮对应的 OnClick 函数里面调用SDK提供的数据发送接口，来发送数据。

可视化埋点只需要研发人员集成采集 SDK，不需要写埋点代码，业务人员就可以通过访问分析平台的“圈选”功能，来“圈”出需要对用户行为进行捕捉的控件，并对该事件进行命名。圈选完毕后，这些配置会同步到各个用户的终端上，由采集 SDK 按照圈选的配置自动进行用户行为数据的采集和发送。

全埋点是通过在产品中嵌入SDK，前端自动采集页面上的全部用户行为事件，上报埋点数据，相当于做了一个统一的埋点。然后再通过界面配置哪些数据需要在系统里面进行分析。

2.3 埋点数据上报时机

埋点数据上报时机包括两种方式。

方式一，在离开该页面时，上传在这个页面产生的所有数据（页面、事件、曝光、错误等）。优点：批处理，减少了服务器接收数据压力。缺点：不是特别及时。

方式二，每个事件、动作、错误等，产生后，立即发送。优点：响应及时。缺点：对服务器接收数据压力比较大。

5.1 采集链路分析

flume采集链路：mock01生成日志文件 –> taidir source –> json拦截器 –> kafka channel –> kafka(log_mock01)
flume消费链路：kafka source –> file channel –> hdfs sink

日志生成的mock01脚本所在服务器是180、181，那么采集日志的脚本也应该在180、181，这里flume的作用是把模拟生成的日志采集到kafka里。exec严格来说是不应该用于生产环境的，spooling directory不能支持文件的更新，只能用taildir source。而且taildir source可以由flume自主控制采集的速度，但是exec、netcat等就不行，channel满了就会回滚，回滚就是清空putlist缓存，就需要source重新采集数据，这种情况对于exec、netcat来说，它是无法重新采集数据的。但是taildir source可以，它基于文件，只要事务不提交，就不会更新offset，就能重新采集。一般是要通过监控等手段保证不能让channel满的。

不管是行为日志还是业务日志，把数据送往kafka都是由180、181来做的，把数据从kafka送往hdfs都是由182来做的。

kafka channel，意思是把数据放入kafka，注意一旦把数据放入kafka，就意味着谁都可以过来消费了，kafka channel的使用场景比较灵活，数据放入kafka，就意味着把数据放入了磁盘中。注意在Flume1.7以前，Kafka Channel很少有人使用，因为发现parseAsFlumeEvent这个配置起不了作用。也就是无论parseAsFlumeEvent配置为true还是false，都会转为Flume Event。这样的话，造成的结果是，会始终都把Flume的headers中的信息混合着内容一起写入Kafka的消息中，这显然不是我所需要的，我只是需要把内容写入即可。

batchSize大小如何设置？Event 1K左右时，500-1000合适（默认为100）

flume的source从上游读数据，不管读到什么数据，都会给包装成event，如果用kafka channel，它就会直接把event而不是真实数据放入kafka，这样下游读到的就都是event对象。真实数据存在event里的body里，下游就必须引用flume的依赖解析这个event对象，把body拿出来才能用这个数据，这样显然是不现实的。所以我们应该把真实数据放入kafka供下游直接读取，而不是把event放进去

channel配置的时候有个参数parseAsFlumeEvent，默认是true，意思是把数据包装成event，改为false，这样就会把真实数据放入channel，这样的结果是event的结构就丢失了。

源数据是json格式，json格式对格式要求很严格，如果数据断了一截，就解析不出来了，而前端采集数据的时候很可能会出问题导致传过来的数据不是json格式，所以在这里可以做个json格式校验，把不是json格式的日志过滤掉，可以通过拦截器做这个操作。

5.6 kafka -> hdfs

好了现在数据已经在kafka了，如何把数据搬运到hdfs呢？不是说必须用flume，它作为一个成熟的日志传输框架，你不用也行，就得写java代码。这里有两种方式可以选择，看起来方式一架构更简单，但实际得选方式二。

零点漂移：hdfs sink可以在写数据的时候滚动目录，可以实现把每天的日志放入一个文件夹，具体目录怎么滚动是按headers里的时间戳来区分的，当时配的是hdfs.useLocalTimeStamp = true，意思是以flume sink的处理时间为准进行滚动。这里就有个零点漂移的问题，某天23:59:50时发生的事件，在flume处理时，时间可能就变成了第二天的00:00:30，显然我们想用的语义是事件时间而不是处理时间。flume会自动识别event的headers里面的timestamp属性，可以设置hdfs.useLocalTimeStamp = false，再次利用拦截器把日志里的timestamp放到event里的headers里面来解决零点漂移的问题，而拦截器必须得有source。

file channel的效率够用吗：生产环境为了安全起见用的都是file channel。file channel的性能大概在20M/s，只要下游能在一天内以20M/s的速度把数据消费完，就不会有问题，这个数据量是很大的，如果实在消费不完，用消费者组。传输的是普通日志信息（京东内部一天丢100万-200万条，这是非常正常的），也可以选择MemoryChannel。也可以通过配置dataDirs指向多个路径，每个路径对应不同的硬盘，来缓解硬盘的写数据压力，从而增大Flume吞吐量。

只有flume才有拦截器吗：不是，任何一个流式框架都是有拦截器的，kafka也有拦截器，它的生产和消费都有拦截器，可以利用consumer的拦截器加时间戳然后再用flume进行sink，这就是方式一了，为什么最终没用方式一呢？因为kafka的拦截器写起代码来极其麻烦。

六、业务数据同步

1. 数据同步方式

数据同步分全量同步和增量同步，周期的话离线一般是天，但是也有些表要求每小时同步一次，然后每小时计算出结果。注意增量同步同步的是新增和变化，它第一次同步的时候得是全量同步。

增量上云的时候不容易拿到最新的数据，比如mid为001的用户名是张三，过两天改成了李四，云上就会既保留张三也保留李四，用的时候想拿最新的数据就不好拿。有个可行的办法是，增量上云，然后在dataphin上给它合并成全量表，结果表里也是每天都存全量的最新数据，注意这种方式是永远无法删除注销数据的，只能再去关联。而增量导出的时候也有个问题，比如今天把001导过去了，过两天001注销了，业务又不想看注销用户的数据，那就要考虑在增量的情况下如何把001删除。像上述两个问题在全量同步的情况下都是不存在的，但是全量同步在数据量大的情况下很慢。

现在存储的代价很低了，无非就是多加几块磁盘，尽量优先全量同步。

注意这里有张加购表，既用了全量同步又用了增量同步，后续会说原因。

2. 数据同步工具

数据同步工具种类繁多，大致可分为两类，一类是以DataX、Sqoop为代表的基于Select查询的离线、批量同步工具，另一类是以Maxwell、Canal为代表的基于数据库数据变更日志的实时流式同步工具(例如MySQL的binlog，其会实时记录所有的insert、update以及delete操作)。全量同步通常使用DataX、Sqoop等基于查询的离线同步工具。而增量同步既可以使用DataX、Sqoop等工具，也可使用Maxwell、Canal等工具，下面对增量同步不同方案进行简要对比。

sqoop的底层原理：需要传数据库的地址用户名密码，然后传给它一个SQL，sqoop拿到这些信息后，会通过jdbc的方式连接数据库，然后执行SQL将所需的数据抽取出来，然后再把数据写入hdfs。它跑的是个没有reduce的MR。

DataX/Sqoop：原理是基于查询，故若想通过select查询获取新增及变化数据，就要求数据表中存在create_time、update_time等字段，然后根据这些字段获取变更数据。由于是离线批量同步，故若一条数据在一天中变化多次，该方案只能获取最后一个状态，中间状态无法获取。

Maxwell/Canal：要求数据库记录变更操作，例如MySQL需开启binlog，由于是实时获取所有的数据变更操作，所以可以获取变更数据的所有中间状态，有时候中间状态还是挺重要的，比如订单的中间状态。

本项目中，全量同步采用DataX，增量同步采用Maxwell。maxwell最大的优点就是部署简单，但是只支持json，灵活性不如canal，详细教程见datax和maxwell篇。

### 3.2 扩展：getopt模块

在运行程序时，可能需要拿到命令行输入的参数，在Python中提供了getopt模块很好的实现了对这种用法的支持，而且使用简单。

3个概念：

  • options：分析出的格式信息，它是一个两元组的列表。每个元素为：(选项串, 附加参数)，比如命令行里自己输入的–help、-v就是选项串，如果没有附加参数，附加参数则为空串’’
  • arguments：不属于格式信息的剩余的命令行参数
  • args：指的是当前脚本接收的参数，它是一个列表，可以通过sys.argv获得

基本介绍：

  • 下面定义了’-h-f:-v’，在-f后面多了一个”:”，这个”:”代表了当前参数是有值的，是一个参数名+参数值的参数
  • 短参数使用的时候是参数名[空格]参数值，比如：-f test
  • 长参数使用的时候是参数名=参数值，比如：–filename=test

4.3 教学版maxwell

仔细观察，会发现目标路径中的日期，并非模拟数据的业务日期，而是当前日期。这是由于Maxwell输出的JSON字符串中的ts字段的值，是数据的变动日期。而真实场景下，数据的业务日期与变动日期应当是一致的。此处为了模拟真实环境，对Maxwell源码进行了改动，增加了一个参数mock_date，该参数的作用就是指定Maxwell输出JSON字符串的ts时间戳的日期，接下来进行测试。

4.5 增量表首日全量同步

通常情况下，增量表需要在首日进行一次全量同步，后续每日再进行增量同步，首日全量同步可以使用Maxwell的bootstrap功能，方便起见，下面编写一个增量表首日全量同步脚本。

编写脚本：mysql_to_kafka_di_init.sh，该脚本的作用是初始化所有的增量表，只需执行一次

coding 
 
    #!/bin/bash
    #####################################################
    MAXWELL_HOME=/opt/module/maxwell-1.29.2
    #####################################################
    import_data() {
        $MAXWELL_HOME/bin/maxwell-bootstrap --database project01 --table $1 --config $MAXWELL_HOME/config.properties
    }
    #####################################################
    case $1 in
    "cart_info")
    import_data cart_info
    ;;
    "comment_info")
    import_data comment_info

    import_data coupon_use

    ;;

    import_data order_info
    ;;
    "order_refund_info")
    import_data order_refund_info
    ;;
    "order_status_log")

    import_data order_detail_coupon
    import_data order_info
    import_data order_refund_info
    import_data order_status_log
    import_data payment_info
    import_data refund_payment
    import_data user_info
    ;;
    esac
